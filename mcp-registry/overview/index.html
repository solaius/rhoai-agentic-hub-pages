What's Missing

There is no single place that tracks what MCP servers exist, what state they are in, which are intended for consumption, or how a deployed instance relates to the asset it was deployed from. The MCP Registry fills this gap by becoming the governed record that other components read from.

The Registry

What the MCP Registry Does

The registry stores metadata-first records for operationalized MCP servers. Each record carries the fields below. Downstream components use lifecycle state to decide what to enforce, surface, or block.

Lifecycle State Governs Operationalized Assets

The registry governs MCPs that have been operationalized — deployed to the cluster or registered as a remote endpoint. Governance begins at that point. Draft and approval states are Catalog-side concerns.

Lifecycle State

Active, deprecated, disabled, or retired. Only active MCPs are eligible for gateway enforcement and Studio consumption.

hosting_model

Hosting Model

Cluster-deployed or remote-registered. Both use one normalized record shape with type-specific metadata fields.

deployment_ref

Deployment / Registration Metadata

Namespace, image reference, deployment association (cluster); endpoint URL, connection details (remote). Core to the record, not optional.

namespace

Namespace / Workspace Scope

Visibility boundary for RBAC-aware access. Users and integrations see only records within their scope.

ⓘ
Mandatory vs. Optional Metadata

Identity fields (name, version, publish state) are mandatory on every record. Extended metadata (description, tags, labels, deployment references) is optional and can be populated incrementally as the record matures.

≠
What the registry does not do

Host MCP runtimes, enforce gateway policies, deploy services, or replace the catalog. It stores the governed state that those components consume.

3.5 Dev Preview Scope

Targeted User Stories

Eleven stories defining the minimum user-visible and system-visible behaviors for the first MCP Registry deliverable. Each story describes who acts, what changes, and which component interaction it implies.

01
Platform Engineer

Register a cluster-deployed MCP server

Create a governed registry record for an MCP deployed to the cluster (Red Hat-provided, customer-created, or third-party MCPs deployed to customer infrastructure).

  • Metadata-first reference, not the runtime itself
  • Canonical MCP payload aligned with upstream MCP registry schema
  • Mandatory: name, version, description, hosting model, lifecycle state
  • Optional: readme, tools list, parameters, runtime prerequisites
Core registry write surface. Governance begins at deployment.
02
Platform Engineer / AI Engineer

Register a remote MCP server

Create a governed registry record for an externally hosted MCP (e.g., GitHub-hosted, SaaS MCP endpoints) without requiring the registry to host the runtime.

  • Stores endpoint URL and connection details
  • Same normalized record shape as cluster-deployed MCPs
  • Version defaults to "latest" when unspecified — declared, not enforced
  • Mandatory and optional metadata same as cluster-deployed path
Components consume one normalized record regardless of hosting model.
03
Platform Engineer / Integration

Retrieve and list MCP records

Retrieve a single record by identifier and version, or list records with structured field filtering.

  • Filter by name (exact or substring), version, lifecycle state, namespace/workspace, owner, hosting model
  • Results filtered by caller's RBAC scope
Core read surface for catalog, gateway, and AAA/Studio.
04
Platform Engineer

Manage operational lifecycle state

Transition MCP records between active, deprecated, disabled, and retired states. Every transition requires a mandatory comment and is logged with user, action, timestamp, and comment.

  • Active: in use, fully operational
  • Deprecated: functional but teams should transition
  • Disabled: blocked from new use; retained for governance
  • Retired: end-of-life; retained for audit only
Components filter on lifecycle state. Draft/approval are Catalog-side concerns.
05
Catalog (Integration)

Provide lifecycle status feedback to the catalog

Catalog queries registry for lifecycle status of operationalized MCPs to enrich catalog discovery with operational state.

  • Catalog queries multiple lifecycle states (active + deprecated) to detect removals and show warnings
  • Discovery precedes governance — Catalog is the discovery surface, Registry is the governance truth
Registry feeds status back to Catalog. Catalog does not define governance state.
06
Platform Engineer

Record deployment and registration metadata

Store deployment or registration details (namespace, image reference, endpoint URL, deployment association) as core registry metadata.

  • Registry does not deploy — lifecycle operator does
  • Deployment/registration metadata are core record fields, not optional extensions
  • In 3.5 DP, associations may be written manually
Bridges "what is governed" and "what is running."
07
MCP Gateway (Integration)

Provide gateway-relevant MCP metadata

Gateway retrieves active MCP metadata for runtime enforcement with sub-200ms lookups.

  • Gateway remains the runtime enforcement layer
  • Registry stores the deployment endpoint (from lifecycle operator); gateway creates its own mediated route
  • Gateway endpoint is a property of gateway configuration, not the registry
  • Full automation is a later-phase goal
Registry becomes governed source for gateway metadata.
08
AAA / Studio (Integration)

Surface operationalized MCPs to AAA / Studio

AAA/Studio queries active, scope-filtered MCPs so AI engineers see only what is intended for consumption.

  • Governed alternative to config-map-driven exposure
Shift from static config to registry-governed consumption.
09
Platform Engineer / AI Engineer

Enforce scoped visibility

Restrict registry views to namespace/workspace-appropriate records via RBAC. Applies to both management and consumption queries.

  • Namespace/workspace-based visibility boundary
  • Does not require a policy engine
All consuming surfaces receive scope-filtered views.
10
Platform Engineer

Preserve version history and manage lifecycle across versions

Retain prior versions with independent lifecycle states for orderly transitions.

  • Each version has its own lifecycle state
  • Consuming components see deprecation/disabled status per version
Catalog and Studio guide users toward current versions.
11
Platform Engineer

Register MCPs deployed outside the catalog path

Register directly-deployed or pre-existing MCPs so all operationalized MCPs are governed, regardless of entry path.

  • MCPs deployed before the registry existed can have records created after the fact
  • Lifecycle operator may write a deployment reference back to the registry after deployment
  • If the MCP originated from the catalog, its catalog metadata (provenance, approval status) may be annotated on the registry record
  • Ensures no deployed MCP server remains ungoverned
Closes the "deploy → never registered" gap. Supports pre-3.5 migrations.

Boundaries

3.5 Dev Preview Scope

For 3.5 DP, the registry stores enough state for components to read from it. Full automation of every handoff is a later-phase goal.

In Scope

  • Register MCP servers (cluster-deployed and remote-registered) as governed records
  • Metadata-first records with mandatory and optional fields
  • Lifecycle state management (active / deprecated / disabled / retired) with audit trail
  • Scoped visibility (namespace/workspace-based, RBAC-enforced)
  • Read access for catalog, gateway, and AAA/Studio integrations
  • Lifecycle status feedback from registry to catalog
  • Deployment and registration metadata as core record fields
  • Version history with independent lifecycle per version
  • Registration of MCPs that bypassed the catalog path, including pre-3.5 deployments

Out of Scope

  • MCP ingestion, curation, scanning, signing, or approval workflows (Catalog-side)
  • Draft or pre-approval states (Catalog-side)
  • Approval workflows, review gates, or policy enforcement engines
  • Full automated lifecycle handoffs between components
  • Detailed API, CRD, or data model design
  • Full multi-asset registry framework
  • Disconnected sync/mirroring
  • UI/UX design
