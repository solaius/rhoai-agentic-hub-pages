GOVERNANCE

MCP Registry

The system of record for MCP servers — governed identity, versioning, and lifecycle state built on MLflow

Identity
Governed Records
  • Stable identity and versioning for every MCP server
  • Canonical server.json aligned with upstream MCP spec
  • Mandatory metadata: name, version, description, hosting model
Hosting
Internal & External
  • Internal: packaged and deployed within the platform
  • External: referenced by remote endpoint URL
  • Same normalized record shape for both
Access
RBAC & Workspace Scoping
  • Users see only MCP records they are entitled to
  • All reads and writes scoped to caller's workspace
  • Unauthorized servers never appear in views
Audit
Auditable State Changes
  • Every lifecycle transition recorded
  • Deployment references associated with registry records
  • Retroactive registration for already-deployed MCPs
Tech Preview · RHOAI 3.5 · 2H 2026
Four key capabilities: Governed Records (identity, versioning, upstream spec alignment), Internal & External hosting support, RBAC & Workspace Scoping, and Auditable State Changes. Built on MLflow upstream. Tech Preview in RHOAI 3.5. Key point: both internal (packaged) and external (endpoint URL) MCPs use the same normalized record shape.
LIFECYCLE

From Draft to Production

Every MCP server follows a governed lifecycle — publish state determines what downstream components can see

Draft
Under review. Not visible to downstream consumers or AI engineers.
→
Active
Published and eligible for Catalog, Gateway, and Studio consumption.
→
Deprecated
Still queryable but not preferred. Steers users toward newer versions.
→
Deleted
Removed from active queries. Preserves audit history.
Platform Engineer View
Registers MCP servers, manages state transitions, associates deployment references. Sees all records in their workspace scope.
AI Engineer View
Sees only Active MCP servers scoped to their workspace in Gen AI Studio. No manual ConfigMap editing, no guessing which servers are available.
Lifecycle states: Draft (RHOAI-specific, not in upstream spec) → Active (maps to upstream "active") → Deprecated → Deleted. Draft is critical for reviewing incoming MCPs before surfacing them. Active is when downstream consumers can see the MCP. Two persona views: Platform Engineer manages the full lifecycle, AI Engineer sees only Active MCPs.
