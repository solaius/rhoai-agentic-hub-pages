Where We Are vs Where We Need to Be

The MCP Onboarding Gap

RHOAI 3.4 ships 10 MCP servers in the catalog (Dev Preview) — 3 Red Hat, 5 partner, and 2 community. Each required manual adaptation. Nothing was automated. This does not scale.

Today: RHOAI 3.4

Manual, per-MCP, per-engineer · 10 MCPs (3 RH + 5 Partner + 2 Community)
  • × MCPs across all sources had different metadata formats — no two containers were structured the same
  • × Manual container adaptation: rebasing to UBI, adding labels, copying licenses manual
  • × Some sources had containers, some had nothing — built from scratch when no Dockerfile existed
  • × CVE scanning via Quay only; dependencies untouched to avoid breaking applications
  • × No end-to-end testing of MCP applications; only container-level checks performed
  • × Onboarding tied to the product release cycle — no way to add MCPs off-cycle
  • × One engineer (Jose) handled all MCP containerization manually

Target: Automated Pipeline

Standardized, repeatable, off-cycle capable
  • ✓ Standardized metadata schema — every MCP container meets a defined checklist automated
  • ✓ Automated containerization with Gen MCP runtime wrapping for auth, TLS, observability
  • ✓ Two ingestion lanes: partner repos (containerized) and customer APIs (Gen MCP generated)
  • ✓ Multi-layer scanning: CVE (Quay), dependency (Dependabot), AI-specific (Snyk) pipeline
  • ✓ MCP Checker evaluations against configurable agents before catalog listing
  • ✓ Partner MCPs managed independently of the release cycle — continuous onboarding and updates
  • ✓ Registry-governed ingestion — provenance, versioning, lifecycle tracking from day one
Business Value

Why This Pipeline Matters

A structured ingestion pipeline unlocks end-to-end ecosystem control and directly addresses what sales, services, and customers are asking for.

End-to-End Ecosystem Control

From ingestion through registry, gateway, and consumption — Red Hat controls the full MCP lifecycle. No gaps, no manual hand-offs, full provenance.

Off-Cycle Partner Onboarding

Partners no longer wait for the next RHOAI release to ship new MCPs. Continuous onboarding means faster ecosystem growth and quicker CVE response.

Customer Self-Service

Platform engineers create MCPs from existing APIs using Gen MCP. Sales teams pitch this capability directly — lower-code path to agentic AI adoption.

Metadata Standardization

Define the Kubernetes MCP container metadata standard before anyone else does. Red Hat leads upstream, partners conform, customers benefit from consistency.

Supply Chain Security

Every MCP entering the platform is validated, scanned, and evaluated before reaching the catalog. Provenance tracked, vulnerabilities caught, trust verified.

Reusable Across Asset Types

Validation, scanning, and evaluation components are designed to extend beyond MCPs — guardrails, agents, and future AI asset types can use the same pipeline.

Ingestion Model

Two Lanes, Shared Pipeline

Partner MCPs and customer-generated MCPs enter through different lanes but share the same validation, scanning, and evaluation infrastructure.

Lane 1

Partner MCP Ingestion

ISV partners provide MCP server repositories that meet a defined checklist. Most arrive with containers; some need production-readiness wrapping.

  • Partner provides repo meeting defined checklist
  • Already-containerized MCPs use podman/docker path
  • Basic MCPs (stdio/HTTP) get Gen MCP runtime wrapping for auth, TLS, observability
  • Pipeline validates, scans, evaluates, and registers
  • Goal: off-cycle onboarding, independent of RHOAI releases
  • Partnership Ecosystem team (Matt Dorn, Jose Gonzalez) manages intake
Lane 2

Customer MCP Generation

Platform engineers create MCPs from existing APIs and repos using Gen MCP. Lower-code approach for rapid MCP creation.

  • API definition (OpenAPI) or HTTP backend as input
  • Gen MCP generates the MCP server code
  • Containerization via KO build tool
  • Auth, TLS, and observability included by default
  • Standardized metadata baked in automatically
  • Sales and services teams actively pitching this capability

Critical Gaps

What Needs to Be Solved

The April 10 meeting identified several gaps that must be addressed before the pipeline can be built.

No Metadata Standard

All 10 MCPs shipping in 3.4 (3 Red Hat, 5 partner, 2 community) had inconsistent metadata. No upstream Kubernetes standard exists for MCP container metadata. Red Hat must define and lead this.

Snyk Capabilities Unknown

Ann Murray recommended Snyk for AI/agent scanning but the exact capabilities and whether it adds value beyond standard CVE scanning needs research.

MLflow + MCP Checker Integration

MLflow is becoming the main evaluation tool for AI assets. How MCP Checker evaluation results feed into MLflow's evaluation framework is undefined.

Pipeline Orchestration Tooling

Konflux was suggested for pipeline orchestration/productization but hasn't been investigated. The build/release pipeline infrastructure is undefined.
