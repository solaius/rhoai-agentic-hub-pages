Top 3 Customer Asks

1. Identity & Token Exchange

Customer Need: "How do I handle identity when my agent calls backend services? How do tokens flow through the stack?"

Status: Shipped MCP Gateway + AuthBridge + Keycloak OIDC + Vault integration addresses this. Token exchange via Keycloak, OBO support via act claim chains, wristband signing for per-tool authorization, Vault secrets injection for backends requiring stored credentials (GitHub PATs, API keys). AuthBridge's current implementation is Kagenti-delivered; Kagenti is being removed from the roadmap as of 2026-07-10, with OpenShell expected to expand to cover this capability. Claim under revalidation.

Evidence: A large enterprise OEM customer validated "dumb MCP server" pattern (May 2026) — MCP server receives validated token in Authorization header, no auth code required. 10 technical Q&As answered covering Okta federation, token caching, OBO configuration, audit trail.

2. Observability & Monitoring

Customer Need: "How do I monitor MCP server usage? Which tools are being invoked? By which agents? What's the latency? What's failing?"

Status: Shipped MCP Gateway provides OpenTelemetry traces, per-tool invocation metrics, structured audit logging, and agent-to-tool correlation. Metrics cover latency, retries, errors, and token usage.

Gap: Planned Cost tracking is not explicitly modeled (strategic hardware partner gap analysis, June 2026). Retry metrics are not differentiated from primary invocations. Cache-aware token rate limiting (cache hit/miss feedback into limiting decisions) is an open question. Planned for GA (Q4 2026).

3. Cost Optimization & Rate Limiting

Customer Need: "How do I control costs when agents fan out to external APIs? How do I prevent runaway usage?"

Status: Partial Rate limiting supported per-tool via Limitador (Red Hat Connectivity Link component). Timeouts and concurrency limits not yet configurable per-tool through MCP Gateway CRDs (strategic hardware partner gap analysis, June 2026).

Roadmap: Full tool-level QoS (timeouts, concurrency, cost budgets) planned for future releases. Envoy provides infrastructure-level timeout configuration today — can be leveraged for near-term POCs even if not yet exposed through MCP Gateway CRDs.

Agent-to-Agent (A2A): Not Yet

A2A Status: Future Work

Agent-to-Agent (A2A) communication is one of the four traffic types AI Gateway must serve (inference, tool calling/MCP, A2A, external provider egress). A2A covers agent discovery (AgentCard), invocation, and orchestration.

Current State: Discussed at AI Gateway F2F (April 2026) but not yet substantively addressed. A2A is planned for future releases but not part of RHOAI 3.5 or MCP ecosystem initial deliverables.

Related Context: The healthcare system's enterprise AI control plane mapping includes "Agent registry and lifecycle management" — overlaps with A2A discovery requirements. The bank's Team B use case (multi-cluster/region/cloud) may require A2A for cross-cluster agent orchestration.

Key Takeaways

What Customers Care About Most

  • Identity is table stakes: Every enterprise customer asks about identity and token flow before anything else. MCP Gateway's auth story (Keycloak + AuthBridge + Vault) is a differentiator.
  • Observability for trust: Customers won't deploy AI agents at scale without visibility into what's happening. OpenTelemetry traces, per-tool metrics, and audit logging are critical.
  • Cost control for safety: Runaway agent usage is a real fear. Rate limiting is a start; cost tracking and budgets are the next ask.
  • Governance before runtime: The telecommunications-provider customer asked about registry early access because they need approval workflows before deploying 20+ MCP servers. Discovery alone isn't enough.
  • Developer experience wins deals: The OEM customer's "dumb MCP server" requirement proves that complexity kills adoption. Gateway-layer auth means developers write business logic, not auth plumbing.
