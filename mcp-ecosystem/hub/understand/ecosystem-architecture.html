Component Roles

MCP Catalog — Discovery

Browse partner, community, and Red Hat MCP servers. Metadata includes tool lists, security indicators (SAST, verified source, read-only tools), license (SPDX), provider logo, README.

Flow: User discovers MCP server → clicks "Deploy" → triggers Lifecycle Operator.

MCP Registry — Governance

System of record for MCP servers. Tracks lifecycle state (Draft → Candidate → Published → Deprecated), approval status, certification, ownership, deployment visibility (is_deployed flag).

Flow: MCP servers from the Catalog are registered INTO the Registry for governance tracking → Registry captures lifecycle state, approval, certification → Registry informs Gateway configuration (auto-register approved MCP servers at GA).

MCP Lifecycle Operator — Deployment

Kubernetes operator watches MCPServer CRDs and creates Deployment + Service + prerequisites (ServiceAccount, ConfigMap, ClusterRoleBinding).

Flow: User clicks "Deploy" in Catalog → Catalog creates MCPServer CR → MCPLO deploys workload → Registry marks is_deployed=true.

MCP Gateway — Runtime Enforcement

API gateway for AI tool calls. Provides virtual MCP servers (aggregate tools from multiple backends), identity-aware tool filtering, per-tool auth, metrics, audit logging.

Flow: Gateway reads registered MCP servers from Registry → auto-creates MCPServer CRDs in gateway namespace → enforces per-tool policies → routes tool calls to backend MCP servers.

Gen AI Studio — Consumption

Playground UI for testing AI models with tool calling. Settings panel shows available MCP servers.

Flow: Studio queries Registry for approved MCP servers → displays in MCP tab → user selects MCP server → model calls tools via Gateway.

The ConfigMap Bridge (Current State)

In RHOAI 3.4, the gen-ai-aa-mcp-servers ConfigMap in redhat-ods-applications namespace is the integration point between deployed MCP servers and Gen AI Studio:

apiVersion: v1
kind: ConfigMap
metadata:
  name: gen-ai-aa-mcp-servers
  namespace: redhat-ods-applications
data:
  OpenShift-MCP: |
    {
      "url": "http://ocp-mcp.my-mcp.svc.cluster.local:8080/mcp",
      "description": "Red Hat OpenShift MCP server"
    }

Current State (3.4): Cluster admins manually edit this ConfigMap after deploying MCP servers. No validation. No approval workflow. No audit trail.

Target State (3.5+): Registry replaces this ConfigMap. Gen AI Studio queries the Registry API for approved MCP servers. ConfigMap becomes read-only (generated from Registry state).

How Components Connect

Catalog → Registry

MCP servers are configured and discovered in the Catalog, then registered INTO the Registry for governance. The Catalog is the source; the Registry receives from it. Registry adds lifecycle state, approval status, certification, and ownership tracking. Registry's governance metadata (e.g., trust tiers) may inform how servers are displayed in the Catalog UI.

Catalog → MCPLO

When a user clicks "Deploy" in Catalog, the Catalog creates an MCPServer CR. MCPLO watches for this CRD and deploys the workload. Deployment state can be reported back to the Registry (is_deployed flag).

Registry → Gateway

Gateway polls Registry for approved MCP servers with lifecycle_state=PUBLISHED. For each, Gateway auto-creates an MCPServer CRD in its namespace, configuring routing and auth policies.

Target (GA): Registry approval → Gateway configuration flows automatically.

Registry → Gen AI Studio

Studio queries Registry API for MCP servers visible to the current user (RBAC-filtered, approval-filtered). Replaces manual ConfigMap editing.

CRDs, ConfigMaps, and APIs

Artifact Owner Purpose
MCPServer (MCPLO) MCP Lifecycle Operator Deployment descriptor (image, ports, secrets, env vars)
MCPServerRegistration (Gateway) MCP Gateway Controller Gateway registration (spec.prefix, targetRef to HTTPRoute), mcp.kuadrant.io/v1
MCPVirtualServer (Gateway) MCP Gateway Controller Curated virtual server composed from one or more registered backends, mcp.kuadrant.io/v1
MCPGatewayExtension MCP Gateway Controller Gateway-level config (trustedHeadersKey, broker settings)
gen-ai-aa-mcp-servers Gen AI Studio (3.4) Manual bridge to Studio; replaced by Registry in 3.5
MLflow Registry API MCP Registry (3.5+) Governed MCP server metadata, lifecycle, approval

Current vs Target State

RHOAI 3.4 (Current — Manual Steps)
  1. Catalog — Browse MCP servers (static metadata, no governance)
  2. Deploy — User creates prerequisites manually (ServiceAccount, ConfigMap, ClusterRoleBinding), then creates MCPServer CR
  3. Register — Cluster admin manually edits gen-ai-aa-mcp-servers ConfigMap
  4. Configure Gateway — If using Gateway, manually create MCPServer CRD in gateway namespace, HTTPRoute, AuthPolicy
  5. Consume — Studio reads from ConfigMap, shows MCP servers in playground settings
RHOAI 3.5+ (Target — Automated Governed Flow)
  1. Catalog — Browse MCP servers with trust tier labels (certified, community, unverified)
  2. Deploy — Click "Deploy" → Catalog creates MCPServer CR → MCPLO creates all prerequisites automatically
  3. Register — Deployed servers registered FROM Catalog INTO Registry for governance (owner, approval, certification, deployment state)
  4. Configure Gateway — Gateway polls Registry, auto-creates MCPServer CRDs for approved servers
  5. Consume — Studio queries Registry API for approved MCP servers (RBAC-aware, no manual ConfigMap)
