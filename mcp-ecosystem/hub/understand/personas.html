Personas

Three key personas interact with the MCP Management ecosystem — each with different roles, goals, and touchpoints.

Platform Engineer

Platform Engineer

Role: Installs and configures the MCP ecosystem infrastructure. Manages cluster-level resources, operators, and namespace boundaries.

Goals:

  • Deploy MCP Gateway, Lifecycle Operator, and Registry with minimal manual steps
  • Configure RBAC policies for who can deploy/register/consume MCP servers
  • Set up namespace isolation (e.g., dev, staging, prod MCP servers)
  • Monitor platform health (operator logs, Gateway metrics, deployment status)
  • Troubleshoot failed deployments or misconfigurations

Components Touched:

  • MCP Gateway: Install via Helm, configure MCPGatewayExtension, create wristband signing keys, set up AuthPolicy for OIDC integration
  • MCP Lifecycle Operator: Install via YAML from release manifest, verify operator pod health, configure namespace labels (opendatahub.io/dashboard)
  • MCP Registry: Configure MLflow backend, set workspace-level RBAC policies
  • OpenShift/Kubernetes: Create ClusterRoleBindings, configure Keycloak realms, set up cert-manager for TLS

Typical Workflow:

  1. Install Tier 1 operators (Service Mesh, cert-manager, RHCL) via OperatorHub
  2. Install Tier 2 components (MCP Gateway, MCPLO, Keycloak) via Helm/YAML
  3. Configure namespace labels and RBAC policies
  4. Create prerequisite resources (ServiceAccounts, ConfigMaps for MCP server configs)
  5. Verify connectivity: curl -X POST http://gateway/mcp -d '{"jsonrpc":"2.0","method":"initialize",...}'
  6. Monitor metrics: OpenTelemetry traces, Prometheus scrapes, Gateway broker logs

Pain Points (3.4):

  • Manual prerequisite creation for every MCP server (ServiceAccount, ConfigMap, ClusterRoleBinding)
  • Namespace label requirement can't be set via oc new-project — requires manifest
  • ConfigMap (gen-ai-aa-mcp-servers) must be manually edited for Studio integration
  • No single "install MCP ecosystem" command — requires orchestrating 10+ YAML applies

3.5+ Improvements:

  • Lifecycle Operator auto-creates prerequisites when deploying from Catalog
  • Registry replaces manual ConfigMap editing — Studio reads from Registry API
  • RHOAI operator bundles MCPLO, reducing manual installation steps

AI Engineer

AI Engineer

Role: Builds AI agents and workflows. Discovers and consumes MCP servers to give agents tool-calling capabilities.

Goals:

  • Discover available MCP servers and understand what tools they expose
  • Deploy MCP servers without needing deep Kubernetes knowledge
  • Test agents with MCP tools in Gen AI Studio playground
  • Build agents in workbenches (Python/Langchain) that call MCP servers
  • Understand which MCP servers are certified/approved vs community/unverified

Components Touched:

  • MCP Catalog: Browse partner, community, and Red Hat MCP servers; read READMEs, view tool lists, check security indicators
  • Gen AI Studio Playground: Select MCP servers in settings panel, test prompts that invoke tools ("list all projects in the cluster")
  • Workbench: Write Python code (Langchain, custom) to call MCP servers remotely (HTTP MCP)
  • MCP Registry (indirect): Benefits from governance metadata — trust tier labels and approval status — that inform what's visible and recommended in the Catalog

Typical Workflow:

  1. Discover: Browse Catalog → find "OpenShift MCP server" → read README for prerequisites and tool list
  2. Deploy: Click "Deploy MCP Server" → select namespace → click Deploy → wait for MCPLO to create resources
  3. Test in Playground: Create playground with tool-calling model (e.g., gpt-4, claude-3.5-sonnet) → Settings > MCP tab → select OpenShift MCP → test prompt: "list all projects in the cluster"
  4. Build Agent: In workbench, write Langchain code that calls http://ocp-mcp.my-mcp.svc.cluster.local:8080/mcp → test locally → deploy as agent image
  5. Monitor Usage: Check Gateway metrics to see how often agent is calling tools, success/failure rates

Pain Points (3.4):

  • No documented way to call MCP servers from Python code in workbenches (Langchain integration gap)
  • Catalog doesn't create prerequisites — must manually create ServiceAccount, ConfigMap, ClusterRoleBinding before deploying
  • Small models (llama-3.2-3b-instruct) struggle with large tool outputs (many pods with many attributes)
  • No visibility into which MCP servers are approved for production vs playground-only

3.5+ Improvements:

  • Trust tier labels in Catalog (certified, community, unverified) informed by Registry governance
  • Workbench templates/examples for calling MCP servers via Langchain
  • Improved model recommendations for tool calling (surface models with strong tool-calling support)

Admin / Governance Lead

Admin / Governance Lead

Role: Approves MCP servers for production use. Sets governance policies (who can deploy what, which tools are allowed). Audits MCP usage for compliance.

Goals:

  • Approve or reject MCP servers for production deployment
  • Set policies: "Agent A can only call Jira read-only tools", "Only certified MCP servers in prod namespace"
  • Audit trail: Who deployed what MCP server? Who called which tools? On whose behalf?
  • Certification visibility: Which MCP servers are certified? Which need re-certification?
  • Risk management: Identify MCP servers with vulnerabilities or expired certifications

Components Touched:

  • MCP Registry: Review pending MCP server registrations → approve/reject → set certification status → revoke approvals if vulnerabilities found
  • MCP Gateway: Configure per-tool AuthPolicies (who can call which tools), review audit logs (full request/response with delegation chains)
  • MCP Catalog (via Registry integration): Indirectly — controls which MCP servers surface in Catalog based on approval/certification status

Typical Workflow:

  1. Review Pending MCP Servers: Log into Registry UI → see list of MCP servers in "Pending" approval status → review metadata (owner, source, tools, security indicators)
  2. Approve for Dev: Click "Approve" → set lifecycle state to "Published" → MCP server now visible in Catalog for dev namespace
  3. Certify for Prod: Run security scan (SAST, vulnerability check) → if passed, set certification status to "Certified" → MCP server now eligible for prod deployment
  4. Set Tool Policies: In Gateway, configure AuthPolicy: groups=["data-science"] can call Jira read-only tools, groups=["admins"] can call all tools
  5. Audit Usage: Query Gateway audit logs for compliance report: "Show all tool calls by user X in last 30 days" → export to SIEM for SOC 2 audit
  6. Revoke Approval: If vulnerability found, set approval status to "Revoked" → Gateway stops routing to that MCP server → Catalog hides it from UI

Pain Points (3.4):

  • No approval workflow — anyone with cluster-admin can edit gen-ai-aa-mcp-servers ConfigMap and add MCP servers to Studio
  • No audit trail for MCP server registration (who added it, when, why)
  • No certification status tracking (which MCP servers are production-ready vs playground-only)
  • No central policy enforcement (can't set "only certified MCP servers in prod namespace")

3.5+ Improvements:

  • Registry provides full governance: lifecycle state, approval status, certification status, ownership, audit trail
  • Gateway integration with Registry enables automatic policy enforcement (approved MCP servers auto-registered in Gateway)
  • RBAC-aware visibility (admins see all MCP servers, AI engineers see only approved)
  • Certification program support (external allowlists, expiration tracking, re-certification workflows)
