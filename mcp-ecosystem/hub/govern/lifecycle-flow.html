Today (3.4) vs Target (3.5+)

Today (3.4)

Stage Mechanism
Discover MCP Catalog UI (manual browsing)
Deploy MCPLO creates Deployment + Service from MCPServer CRD
Register Manual — Admin registers server with Gateway via CRD
Configure Manual ConfigMap — gen-ai-aa-mcp-servers in redhat-ods-applications
Consume Gen AI Studio reads ConfigMap, exposes tools to playground
Monitor Per-server logs, no unified observability
Retire Manual deletion of MCPServer CRD, manual ConfigMap cleanup

Target (3.5+)

Stage Mechanism
Discover MCP Catalog UI (enhanced with trust tiers, verification status)
Deploy MCPLO creates Deployment + Service + auto-registers in Registry
Register Auto — Registry creates governed record with approval workflow
Configure Auto — Registry approval flows into Gateway configuration
Consume Gen AI Studio reads from Registry API, no ConfigMap needed
Monitor Unified observability: per-tool metrics, health status, usage analytics
Retire Registry lifecycle state → Deprecated → Retired (auto-revokes Gateway access)
Key Difference: Today, each handoff (Deploy → Register → Configure) requires manual admin intervention. Target state: Registry is the single source of truth that drives automation across the ecosystem.

Governance Tracks (Current Data Model)

The MCP Registry's current data model uses two independent enums, not the four-track governance model described in early design proposals:

Enum States Scope
MCPServerStatus ACTIVE, DEPRECATED, RETIRED Server-level (3 states)
MCPPublishState DRAFT, PUBLISHED, DEPRECATED, RETIRED Version-level (4 states)
Post-MVP phasing: No approval_status, verification_status, or certification_status field exists in the current model. The four-track governance model (Lifecycle State: Draft/Candidate/Published/Deprecated/Retired; Approval Status: Draft/Pending/Approved/Rejected/Revoked; Verification Status: Unverified/Verified; Certification Status: None/Candidate/Certified/Expired/Revoked) is a proposal, deferred post-MVP, not abandoned: "For MVP/DP we should follow the same path as they are taking for the skills registry upstream to speed up approval/implementation... we should follow immediately after with the governance implementation as proposed here" (doc comment thread). See MLflow MCP Registry Data Model Proposal for the full four-track proposal.

Governance Invariants

  • MCPServerStatus and MCPPublishState are independent enums; no approval, verification, or certification field exists in the current model to gate either transition
  • workspace on a version resolves to the same value as the parent MCPServer.workspace
  • update_mode (MANUAL or SYSTEM_MANAGED) controls whether a controller such as MCPLO may write to the record; it does not itself change lifecycle or publish state
