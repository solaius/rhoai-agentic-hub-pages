Component Integration

How each component in the MCP ecosystem communicates with the others, and how integration evolves from manual configuration to registry-driven automation.

Integration Flows

Flow 1: Catalog → MCPLO (Automated Today)

User browses Catalog
Discovers OpenShift MCP
→
Clicks "Deploy"
Selects namespace
→
Catalog creates MCPServer CRD
YAML applied to cluster
→
MCPLO watches CRD
Creates Deployment + Service

Flow 2: MCPLO → Gateway (Manual Today, Auto Target)

Today (3.4) — Manual

  1. MCPLO deploys MCP server (Pod + Service created)
  2. Admin manually registers server with Gateway via MCPServer CRD
  3. Admin manually updates gen-ai-aa-mcp-servers ConfigMap with server URL
  4. Requires cluster-admin privileges for ConfigMap patch

Target (GA Q4 2026) — Auto

  1. MCPLO deploys MCP server (Pod + Service created)
  2. MCPLO auto-registers server in Registry as "discovered" entry
  3. Approver reviews in Registry UI, approves for production use
  4. Registry auto-configures Gateway routing and auth policies
  5. No manual ConfigMap, no cluster-admin required

Flow 3: Registry → Gateway (Planned GA Q4 2026)

MCP server registered
In Registry (manual or auto)
→
Approval workflow
Governance review + approval
→
Registry updates Gateway
Auto-config MCPGatewayExtension
→
Gateway routes traffic
Tools visible to Studio
Key Principle: Flow is always Catalog → Registry, never reverse. The Registry is the authoritative governance layer; the Catalog is the discovery layer.

Flow 4: Catalog → Registry (Planned 3.5 TP)

MCP servers configured in the Catalog can be registered INTO the Registry for governance:

  • Registration — Catalog-discovered servers are submitted to Registry for governance tracking
  • Governance metadata — Registry adds lifecycle state, approval status, certification, ownership
  • Trust tiers — Registry's certification status may inform how servers are displayed back in the Catalog UI (certified, community, unverified)
  • Deprecation tracking — Registry lifecycle state (deprecated/retired) can signal servers that should be removed from active use

Flow 5: Studio → Gateway/Servers (Manual Bridge Today)

The gen-ai-aa-mcp-servers ConfigMap is the critical integration point:

apiVersion: v1
kind: ConfigMap
metadata:
  name: gen-ai-aa-mcp-servers
  namespace: redhat-ods-applications
data:
  OpenShift-MCP: |
    {
      "url": "http://ocp-mcp.my-mcp.svc.cluster.local:8080/mcp",
      "description": "Red Hat OpenShift MCP server"
    }

In 3.5, this ConfigMap will be replaced by direct Registry API consumption — Studio will query the Registry for approved MCP servers and their endpoints.
