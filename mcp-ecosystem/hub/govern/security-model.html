Defense in Depth: Multi-layer security from Keycloak (identity) → Authorino (policy) → Wristband (per-tool auth) → RBAC (namespace scoping) → Vault (credential brokering).

Authentication Flow

1. User Authentication (Keycloak)

Keycloak sits at the center of the ecosystem as the identity provider:

  • Identity Federation — Keycloak acts as broker, delegating to OpenShift OAuth
  • MCP Realm — Dedicated realm with realm roles (user, admin)
  • Two OIDC Clients:
    • Confidential client for Gateway broker (service-to-service with client secret)
    • Public client for Gen AI Studio/Playground (authorization code flow, no secret)

2. Policy Enforcement (Authorino)

Authorino enforces policy at the Gateway level via AuthPolicy CRD:

  • JWT Authentication — Validates Bearer tokens against Keycloak OIDC
  • Wristband Signing — OPA Rego extracts per-server tool roles from JWT resource_access claim, signs into ES256 wristband JWT in x-authorized-tools header
  • VirtualMCPServer Routing — CEL reads groups claim, sets x-mcp-virtualserver header

3. Wristband Tokens (Per-Tool Auth)

Wristband tokens provide fine-grained per-tool authorization:

  • Signed by Authorino using ECDSA private key
  • Verified by Gateway broker using public key (referenced in MCPGatewayExtension.spec.trustedHeadersKey)
  • Contains per-server tool roles extracted from Keycloak JWT
  • Passed in x-authorized-tools header

4. RBAC (Namespace Scoping)

Kubernetes RBAC controls deployment and management access:

  • ServiceAccount + ClusterRoleBinding — MCP servers run with restricted cluster-wide read access
  • Namespace Labels — opendatahub.io/dashboard: "true" required for RHOAI visibility
  • ReferenceGrants — Cross-namespace references for MCPGatewayExtension → Gateway

Credential Brokering (Vault Integration)

For MCP servers requiring per-user credentials (e.g., GitHub PATs):

Vault Setup

  1. Enable KV v2 secrets engine
  2. Configure JWT auth method with Keycloak JWKS
  3. Create templated policy scoping read access to secret/data/mcp-gateway/users/{{username}}/*

Per-Server AuthPolicy Wiring

  1. Extract Keycloak JWT from Authorization header
  2. Exchange JWT for Vault token
  3. Retrieve credential from Vault KV
  4. Inject credential into Authorization header for MCP server
Known Gap: Vault integration is optional for MVP but required for production use cases with external MCP servers (GitHub, Slack, etc.).

Gateway Topology Options

Topology When to Use
Shared Gateway Policy-isolated teams with centralized governance. Single Gateway instance with per-team VirtualMCPServers and AuthPolicies.
Gateway-per-Team Namespace-isolated teams with strict compliance requirements. Each team deploys its own Gateway in its own namespace.

Key Security Recommendations

  • HTTP-based MCP servers only — stdio transport not supported by Gateway
  • Wristband + VirtualMCPServer layering — defense-in-depth approach
  • Gateway-level AuthPolicy for auth/wristband; per-server HTTPRoute for credential injection
  • Prefer OpenShift OAuth federation over seed users in production Keycloak
  • Namespace isolation for multi-tenant deployments
