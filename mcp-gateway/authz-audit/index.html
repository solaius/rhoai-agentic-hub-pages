THE PROBLEM

The Authorization Gap

Most agentic AI platforms solve authentication — proving an agent's identity. But authentication alone does not answer the harder questions: what can this agent do?

The Authorization Gap
The absence of an enforcement layer between "this agent has a valid token" and "this agent is authorized to perform this specific action." Without it, any authenticated agent can call any tool, access any data, and perform any operation — with no way to prove what happened after the fact.
WITHOUT AUTHORIZATION
Agent + Valid Token = Full Access
  • Agent calls transfer_funds — succeeds
  • Agent queries 10,000 customer records — succeeds
  • No audit trail of what happened or who authorized it
WITH MCP GATEWAY AUTHORIZATION
Agent + Valid Token + Policy = Scoped Access
  • Agent calls transfer_funds — 401 Denied (no role mapping)
  • Agent queries customers — scoped to its tenant's records only
  • Every action logged with user, agent, tool, timestamp, and decision

Why This Matters for Regulated Industries

FINANCIAL SERVICES

GOVERNMENT

ENTERPRISE SaaS

Key Concepts

Available MCP Gateway
An Envoy-based gateway for Model Context Protocol (MCP) servers. Provides runtime connectivity, enforcement, and observability for MCP traffic. Built by the Kuadrant team, ships as part of Red Hat Connectivity Link (RHCL).
Available Authorino
The authentication and authorization engine within RHCL. Evaluates identity, access policies, and token claims as an Envoy ext_authz filter. This is where per-tool authorization decisions are made.
Available AuthBridge
A token exchange proxy (part of Kagenti) injected as an Envoy sidecar. Transparently handles RFC 8693 token exchange so MCP servers never see raw user credentials. Includes in-process token caching.
Available RHCL (Red Hat Connectivity Link)
The connectivity management platform underlying MCP Gateway. Implemented as a WASM plugin for Envoy — not a standalone gateway. This means organizations using OpenShift Service Mesh or Istio do not need major infrastructure changes.
AUTHORIZATION MODEL

Two-Gate Authorization

Every request passes through two policy decision points (PDPs) before reaching the MCP server. Both are enforced at the gateway — the MCP server receives only pre-authorized requests.

The MCP server stays "dumb." AuthBridge transparently offloads all auth so the MCP server receives a validated token in its Authorization header. No token exchange, no credential management, and no auth code is required in the MCP server itself. The Envoy sidecar handles JWT validation transparently.

TOKEN EXCHANGE

RFC 8693 Two-Hop Token Exchange

No token passthrough. Every hop gets a fresh, scoped token. The act (actor) claim chain grows at each hop — creating a cryptographic delegation trail.

No Token Passthrough
Every hop gets a fresh, scoped token. If a jailbroken agent tries to reuse a user's token directly, the gateway rejects it because the token audience and scope don't match.
Token Caching
AuthBridge caches exchanged tokens in-process (SHA-256 keyed, 10K entries, 30-second early-refresh buffer). Consecutive calls don't re-exchange — the cached token is served until TTL expiry.
AGENT IDENTITY

Workload Identity for AI Agents

Agents get their own cryptographic identity — not inherited from pod service accounts. No static credentials. Auto-rotating certificates. Full zero-trust alignment.

Available TIER 1
OpenShell Token Masking
  • Prevents agents from seeing raw user credentials
  • Uses HTTP_PROXY/HTTPS_PROXY env vars
  • Can run centrally outside individual pods
  • Simple to adopt — minimal infrastructure change
Available TIER 2
SPIFFE/SPIRE + AuthBridge
  • Each agent gets its own SPIFFE SVID (X.509 certificate)
  • Auto-rotating — no static credentials to manage
  • Full actor-chain tracking via nested act claims
  • OPA policy enforcement across the delegation chain
  • Kagenti auto-registers agents with Keycloak via SPIFFE ID

Privilege escalation prevention: When Agent A calls Agent B, both agents' SPIFFE IDs are stored in the token's act claim chain. OPA enforces the lowest permission level across the entire chain. Example: if a developer gives an AI coding assistant an HR access token, the coding assistant's workload identity means the system can deny access even with the valid token — because the agent's identity doesn't have HR permissions.

Enterprise IdP Federation
Keycloak identity-brokers with enterprise IdPs (Okta, EntraID, PingIdentity) via OIDC federation. Protocol mappers translate group claims from the source IdP token to the Keycloak-minted token. Sync timing is configurable: sync at first login only, or sync every authentication. No LDAP sync required.
DEFENSE IN DEPTH

Five Authorization Layers

Authorization is not a single check — it's a stack of independent layers, each catching what the others miss. From "can this agent call this tool?" to "are these arguments policy-compliant?"

1
Agent-to-Tool Authorization Available
Can this agent call this tool? — Identity-aware tool filtering at the MCP Gateway. Allow/deny lists per tool, per identity. Enforced via Authorino before the request reaches the MCP server. Example: Customer support agent can call search but not transfer_funds.
2
Agent-to-Data Authorization Available
Can this agent access this resource? — Group-based tenancy generalized across MCP tool catalogs, vector stores, file storage, and guardrails. Each microservice performs its own authorization checks; the gateway establishes identity via headers. Gateway header stripping prevents spoofing.
3
Delegation Authorization Available
Is the agent acting within the user's scoped permissions? — RFC 8693 token exchange + nested act claim chain. OPA enforces the lowest permission level across the delegation chain. Prevents privilege escalation even when agents chain together.
4
Tenant Boundary Enforcement Available
Are tenant isolation boundaries maintained? — Cross-tenant traffic denied by default via NetworkPolicy. Multiple gateway instances per cluster for hard tenant segregation. Explicit admin authorization required for any cross-tenant access.
5
Argument-Level Validation Planned
Do the tool-call arguments comply with business policy? — Even an authorized agent calling an authorized tool can violate policy by passing bad arguments. Example: a refund agent calling process_refund with an amount exceeding the $500 limit, or targeting an ineligible order. Action rails validate arguments against configurable business rules before the call executes.

Additional Guardrail Layers Planned

Trajectory Rails
Detect suspicious patterns across sequences of tool calls — not just individual calls. Catches multi-step attacks that look innocent individually.
Budget Rails
Per-session limits on token usage, tool-call count, and tool-call rate. Enforced before each tool call to prevent runaway agents.
Intent-Alignment Rails
Block agent tool calls that contradict the user's original intent. Prevents agents from drifting beyond their assigned task scope.
CREDENTIAL MANAGEMENT

Three-Tier Credential Strategy

Different backends need different credential patterns. The MCP Gateway supports three tiers — choosing the right one for each backend service while keeping MCP servers credential-free.

Tech Preview
Tier 1: HashiCorp Vault
STORED CREDENTIALS
For backends that don't support OAuth2 natively. Per-user or per-service PATs and API keys stored securely in Vault and injected at runtime.
USE CASES
GitHub PATs, Jira API tokens, custom API keys
Available
Tier 2: OAuth2 Token Exchange
PRIMARY FLOW — NO STORED CREDENTIALS
RFC 8693 token exchange via AuthBridge. Tokens are exchanged and scoped at each hop. No credentials stored — the token itself carries the authorization. This is the recommended approach.
USE CASES
Any OAuth2-capable backend, internal microservices
Available
Tier 3: Direct Token Passing
SAME TRUST DOMAIN
For internal services within the same Keycloak realm. The gateway-scoped token is passed directly — no additional exchange needed. Fastest path, lowest latency.
USE CASES
Internal RHOAI services, same-cluster MCP servers

Known Gap Centralized Audit Log Aggregation
Today, the full audit trail is reconstructed from Keycloak events + gateway logs + MCP server logs. A centralized view that correlates all four layers into a single audit query interface is not yet available. Each layer produces its own logs and events, and correlation is done at the log aggregation layer (e.g., OpenShift Logging, Grafana, Splunk). A unified audit API is on the roadmap.

The act Claim: Cryptographic Proof of Delegation

The nested act claim in the JWT is the cornerstone of the audit trail. It provides a cryptographically signed record of every entity in the delegation chain — from the original user, through each agent, to the backend service. This is not a log entry that can be tampered with; it's embedded in the token itself and signed by Keycloak.
