Value Proposition

How Red Hat MCP Gateway solves the enterprise problems that the MCP protocol alone cannot.

The Enterprise Pitch

"Your agents need governed access to tools, not a free-for-all."

The Model Context Protocol (MCP) is only 16 months old. It defines a clean interface for agent-to-tool communication, but it deliberately leaves critical enterprise concerns unaddressed: authentication, authorization, multiplexing, observability, and governance. MCP assumes these capabilities live in the platform layer.

Red Hat MCP Gateway is that platform layer. It extends MCP with production-grade infrastructure for enterprises who need more than just a protocol — they need governed, observable, secure access at scale.

Three Core Problems We Solve

Problem: MCP has no authentication or authorization

The MCP specification deliberately omits authentication and authorization. MCP servers run without identity awareness, credential brokering, or per-tool access control. In practice, this means agents have unrestricted access to every tool an MCP server exposes — no way to enforce "this agent can only invoke these three tools."

Solution: Tool-level auth via Authorino + Keycloak + AuthBridge

Red Hat MCP Gateway uses Authorino (policy-driven auth engine), Keycloak (enterprise identity), and AuthBridge (token exchange) to provide:

  • Identity-aware filtering: Agents see only the tools they're authorized to invoke
  • Per-tool allowlisting: Deny all by default, grant per-agent, per-user, per-group
  • OAuth 2.0 credential brokering: Token exchange for Jira, GitHub, Salesforce — backends receive valid tokens in their Authorization headers
  • On-Behalf-Of (OBO) support: Full delegation chain tracking via act claim

Problem: MCP has no multiplexing or aggregation

Each MCP server runs independently with its own endpoint and tool namespace. An enterprise with 20+ MCP servers would need to configure 20+ separate connections in every agent. Tool name collisions are unmanaged — if two servers both expose search, there's no way to distinguish them without manual namespacing.

Solution: Virtual MCP servers + tool prefixing

The MCP Gateway Broker aggregates multiple backend MCP servers into a single virtual MCP endpoint at /mcp. Agents connect once and see all tools they're authorized for. Tool prefixing prevents collisions — jira_search vs github_search.

  • Single /mcp endpoint for agents — no multi-connection configuration
  • Automatic tool prefixing based on the MCPServerRegistration CRD prefix field
  • Hot-reload when new MCP servers register — no agent restart required
  • Identity-scoped views: Different agents see different tool catalogs based on their permissions

Problem: MCP has no governance or observability

MCP servers don't track which agent invoked which tool, don't meter token usage, and don't enforce rate limits. There's no audit trail, no cost attribution, and no way to enforce quotas. In production, this means no compliance, no accountability, and no cost control.

Solution: Identity-aware filtering + per-tool metrics + policy-driven routing

The gateway provides full observability and governance:

  • Structured audit logging: Every tool invocation logged with agent ID, user ID, tool name, timestamp
  • Per-tool invocation metrics: Prometheus metrics with tool-level granularity
  • OpenTelemetry traces: Agent-to-tool correlation via trace spans
  • Rate limiting via Limitador: Per-tool, per-agent, per-user rate limits
  • Token metrics: Track tokens consumed per tool invocation (when MCP servers report usage)

Key Differentiators

  • Tool-level authentication and authorization — not just API-level like traditional gateways
  • Credential brokering with token exchange — agents never see raw user credentials
  • Per-tool metrics and audit logging — full observability with tool-level granularity
  • Policy-driven routing — identity-aware filtering and dynamic backend selection
  • Virtual MCP servers — single endpoint, aggregated tool catalogs
  • Multi-cluster support via RHCL — HA/DR across data centers
  • Open source upstream (Kuadrant) — no vendor lock-in
