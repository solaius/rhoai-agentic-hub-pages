Large Enterprise OEM Customer — Security Deep-Dive

Meeting Date: May 2026
Participants: Customer security and engineering leads (OEM customer); Red Hat account and engineering team

Focus: Authentication Architecture

The customer's MCP security discussion centered on one critical question: Can we build "dumb" MCP servers that offload all authentication and authorization to the gateway?

The customer uses Okta as their IdP and needs to federate identity into Keycloak for multi-tenant authorization. Their MCP servers must receive validated tokens without implementing any auth logic themselves.

10 Security Questions Answered

The customer asked 10 detailed security questions covering:

  • Okta group mapping into Keycloak (OIDC federation + protocol mappers)
  • External Keycloak support (confirmed — no co-location requirement)
  • Token flow to backends like Jira and GitHub (OAuth2 token exchange)
  • Token caching implementation (in-process memory, 10K entries, SHA-256 keyed)
  • Consecutive call behavior (cached, no re-exchange on repeated calls to same audience)
  • On-Behalf-Of (OBO) configuration (token exchange with scoped clients)
  • Audit trail (Keycloak events + act claim chain)

Key Outcome: "Dumb MCP Server" Validated

Customer security lead (OEM customer, May 2026):
"I just want it to get the access token in its header and not have to worry about anything else."

Red Hat Kagenti Team (2026-05-06):
"AuthBridge is designed to transparently handle offloading the majority of the auth work so that the MCP server can be quite dumb."

Result: The customer's requirement confirmed as supported. MCP servers receive validated tokens in their Authorization headers. No token exchange, no credential management, and no auth code required in the MCP server itself. The Envoy sidecar (injected by Kagenti) handles JWT validation transparently.

Okta Integration Confirmed

The customer's Okta integration uses OIDC federation (not LDAP sync). Keycloak identity-brokers with Okta via OIDC, with protocol mappers to map group claims from the source Okta token to the Keycloak-minted token. Nested groups are traditionally flattened when presented; reconstructing a hierarchy is possible but requires custom work.

POC Status

Status: Pending customer internal approval (eval licenses being prepared as of May 2026)
Timeline: ~2 weeks for approvals (target: week after Red Hat Summit 2026)

Key Takeaway

The customer validated the "dumb MCP server" architecture end-to-end. AuthBridge offloads all auth work to the gateway, enabling MCP servers to focus solely on tool logic. This is the reference architecture for enterprise MCP deployments with external IdPs like Okta.

Major Telecommunications Provider — Platform APIs POC

Meeting Date: May 2026
Participants: Customer network engineering and platform leads; Red Hat account and engineering team

Focus: Multi-Team Internal MCP Platform

The customer's platform-APIs team is building an internal MCP platform for multiple teams within the organization. The goal: stand up a centralized MCP Gateway that aggregates 20+ MCP servers and provides governed access to tools across the organization.

Multiple Teams Interested

  • Network Planning team — lead use case
  • Their central IT organization — infrastructure platform for IT workloads
  • Their external AI-monetization program — exploring MCP for customer-facing APIs

Customer lead (telecommunications provider, May 2026):
"I am the MCP person in this company... I want to stand all this up and get some of these MCP things running."

Deployment in Real Environment

The customer wants to deploy the MCP Gateway in their own internal environment with real traffic — not a demo sandbox. This validates the gateway's production readiness and provides Red Hat with real-world feedback.

Red Hat engineering (May 2026):
"Gateway can be deployed anywhere and it is in a technical preview state. So it actually comes with support... we're marching towards a GA later this year."

Red Hat engineering (May 2026):
"From a gateway perspective, we actually want to be able to deploy it in your local environment because you get a real feel for the product instead of doing it in some demo environment where everything's optimal anyways."

Use Cases

The customer is evaluating two primary use cases:

  1. External monetization — expose internal APIs via MCP for external partners
  2. Internal automation — reduce tickets, speed up triage, automate onboarding

Red Hat offered to collaborate on use case ideation — suggesting monitoring dashboards (Victoria Metrics MCP servers), Ansible automation MCP servers, and root-cause analysis from logs.

POC Status

Timeline: 2-week POC (as of May 2026)
Environment: OCP 4.20 upgrade pending; Red Hat providing AI clusters
Support: Red Hat engineers available for troubleshooting (direct engineer access offered)

Key Takeaway

The customer is building a multi-team MCP platform with 20+ MCP servers. This validates the virtual MCP server architecture and confirms enterprise demand for centralized MCP governance across large organizations.
