RHCL Context

Red Hat Connectivity Link (RHCL) is the connectivity management platform that MCP Gateway runs on — providing authentication, rate limiting, DNS, TLS, and multi-cluster orchestration infrastructure.

What Is Red Hat Connectivity Link?

RHCL is a connectivity management platform built on the upstream Kuadrant project. It provides enterprise-grade API management capabilities for Kubernetes-based applications.

Think of RHCL like a universal adapter for APIs: Just as a universal power adapter lets you plug your laptop into any outlet worldwide (handling voltage conversion, plug shapes, surge protection transparently), RHCL lets you expose APIs to any client (handling auth, rate limits, DNS routing, TLS certificates transparently).

RHCL's three pillars:

  • Policy Management: Manage policies across namespaces, clusters, availability zones, and clouds
  • API Management: Authentication, logging, rate limiting, monitoring, reporting, transformation
  • AI Gateway: Governance, optimization, communication, security, resiliency for AI models and agents

MCP Gateway fits into the third pillar (AI Gateway) but leverages the first two pillars for its policy and API management foundations.

RHCL Component Stack

RHCL is not a monolithic product — it's an orchestrated stack of specialized components:

Component Role Key CRD
Kuadrant Operator Orchestrates all RHCL components, reconciles policy CRDs Kuadrant
Authorino Authentication and authorization engine (OAuth, OIDC, API keys, OPA policies) AuthPolicy
Limitador Rate limiting engine (request/second, token/minute, custom limits) RateLimitPolicy
Cert Manager TLS certificate management (Let's Encrypt, custom CAs) Certificate
DNS Operator DNS integration (AWS Route53, GCP Cloud DNS, Azure DNS) DNSRecord
Policy APIs Unified policy interface across TLS, Auth, DNS, Rate Limits TLSPolicy, DNSPolicy, etc.

Each component is responsible for a specific concern. The Kuadrant Operator stitches them together, ensuring policies are applied consistently.

Communication Path: Request to Response

Let's trace a request through the full RHCL stack (with MCP Gateway in the middle):

Gateway API
K8s standard

Envoy

Rate limiting

Auth engine
KeyCloak
IdP
MCP Gateway
Router + Broker

Backend service

Step-by-step:

  1. Gateway API: Request arrives at Kubernetes Gateway resource (e.g., Gateway with HTTPRoute)
  2. Ingress Gateway (Envoy): Entry point for external traffic; applies TLS termination
  3. Limitador: Checks rate limits (e.g., "100 requests/second per user"); rejects if exceeded
  4. Authorino: Validates OAuth token, checks permissions via AuthPolicy
  5. KeyCloak: Identity provider consulted by Authorino for token validation (if using Keycloak as IdP)
  6. MCP Gateway: Parses MCP protocol, routes to correct backend MCP server
  7. MCP Server: Executes tool logic, returns result

With Service Mesh (Istio/OpenShift Service Mesh), the Istio Control Plane is added to manage sidecar proxies for east-west traffic (service-to-service communication).

Why WASM Matters

RHCL is implemented as a WASM (WebAssembly) plug-in for Envoy, not a standalone gateway product. This design choice has three critical advantages:

  1. Minimal disruption to existing infrastructure

    Organizations already running Envoy for ingress (via OpenShift Service Mesh, Istio, or standalone Envoy) don't need to rip-and-replace their networking stack. RHCL components slot in as WASM plugins, reusing existing Envoy deployments.

  2. Gateway API standardization

    RHCL uses the Kubernetes Gateway API (not the older Ingress API). Gateway API is a role-oriented, portable, expressive routing standard that's becoming the de facto standard for Kubernetes networking. This means RHCL-configured resources are portable across different Kubernetes distributions and cloud providers.

  3. Multi-cluster resiliency

    RHCL supports configuration mirroring across clusters, BYOP (Bring Your Own Policies), and HA/DR/GLB (High Availability / Disaster Recovery / Global Load Balancing). This allows enterprises to deploy MCP Gateway across multiple clusters or clouds with consistent policies.

OpenShift Version Support Matrix

RHCL requires Gateway API support, which became GA in OpenShift 4.19. However, RHCL and RHOAI version support doesn't always align perfectly:

OCP Version Status RHCL Support Notes
OCP 4.19 Full Support + Maintenance Untested Gateway API GA. RHOAI 3.5 targets 4.19, but RHCL 1.4 likely won't support it. Best-effort only.
OCP 4.20 Released Oct 21, 2025 Supported Recommended minimum for RHOAI + RHCL deployments
OCP 4.21 Released Feb 3, 2026 Supported Fully supported
OCP 4.22 May/Jun 2026 v1.4 RHCL 1.4 adds OCP 4.22 support (June 2026)
Known Gap: August 2026

RHOAI 3.5 (August 2026) will support OCP 4.19, but RHCL 1.4 will likely not. This means the configuration (RHOAI 3.5 + RHCL + OCP 4.19) is untested and best-effort support only.

OCP 4.19 maintenance support ends December 2026 (only 4 months of overlap with RHOAI 3.5 TP). Customers are advised to upgrade to OCP 4.20+ for full support.

Mitigation: customers can also remain on RHCL 1.3.x, which supports OCP 4.19, instead of upgrading to RHCL 1.4: this avoids the gap entirely. Upgrading to OCP 4.20+ remains the full-support path.

Deployment Models

RHCL (and MCP Gateway) can be deployed in several configurations depending on entitlement and use case:

OCP 5.0 Platform

MCP Gateway available via catalog (layered installation). OCP 5.0 includes entitlement to MCP Gateway for platform-internal MCP servers. MCP Gateway Operator declares dependency on RHCL Operator. Zero-cost SKU covers this use case.

Risk: OCP 5.0 ships with OLM 1.0, which drops operator dependency declarations entirely. This auto-install pattern (the MCP Gateway Operator declaring a dependency on the RHCL Operator) is flagged internally as likely at risk under OLM 1.0, pending a new strategy.

RHOAI Deployment

Gets zero-cost RHCL SKU entitlement (includes MCP Gateway). MCP auth via Authorino and AuthPolicy. RHCL rate limiting or token rate limiting available.

The MCP Gateway Operator itself is a customer-installed external prerequisite: per RHOAI platform policy (architecture overlay 0008, ODH-ADR-0007), RHOAI documents this dependency and surfaces its absence via a status condition, rather than bundling, auto-installing, or lifecycle-managing the operator.

Entitlement scope: RHCL has moved from the OpenShift business unit to the AI business unit. The expected direction (not yet fully confirmed) is that MCP Gateway shares the same entitlements as Models-as-a-Service (MaaS); rate limiting for AI workloads and Authorino remain covered under the existing agreement. This MaaS parity point is pending confirmation.

Full RHCL Purchase

Full RHCL feature set including MCP Gateway. Additional capabilities: DNS management, HA/DR/GLB (High Availability, Disaster Recovery, Global Load Balancing), advanced rate limiting policies.

For detailed SKU pricing and entitlement matrix, see Plan It > SKUs & Entitlement.
