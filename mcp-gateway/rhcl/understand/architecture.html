Architecture

MCP Gateway is built as a single binary with three core components — Broker, Router, and Controller — that work together to aggregate, route, and manage MCP tool traffic.

Broker — The Aggregator

What it does: Acts as the central MCP endpoint that agents call. It aggregates tools from multiple registered MCP servers and presents them as a unified "virtual MCP server."

Think of it like: A library catalog system. Instead of walking to each bookshelf (backend MCP server) to see what's available, you query the catalog (Broker) once and it shows you everything across all shelves.

How it works: When an agent calls /mcp with tools/list, the Broker aggregates tool lists from all configured backend MCP servers, applies identity-based filtering (hiding tools the agent isn't allowed to see), and returns the combined result.

Router — The Traffic Director

What it does: Parses incoming MCP requests, extracts the tool name from the JSON-RPC payload, and sets HTTP headers that force Envoy to route the request to the correct backend MCP server.

Think of it like: A mail sorting facility. Letters (requests) arrive with addresses written inside the envelope (JSON payload). The Router reads the address and writes it on the outside of the envelope (HTTP headers) so the delivery system (Envoy) knows where to send it.

How it works: Implemented as an Envoy ext_proc (external processing) filter. When a request arrives at /mcp, Envoy pauses and asks the Router: "Where should this go?" The Router parses the MCP protocol, identifies the target tool (e.g., jira/create_issue), looks up which backend serves that tool prefix, and sets headers like X-MCP-Target-Server: jira-backend. Envoy then routes accordingly.

Controller — The Discovery Engine

What it does: Watches Kubernetes resources (specifically HTTPRoute and MCPServer CRDs) to discover new MCP servers and dynamically update the Broker and Router configurations.

Think of it like: A real-time inventory manager. It watches the warehouse (Kubernetes cluster) for new shipments (MCP server deployments) and automatically updates the catalog (Broker config) and shipping routes (Router config) without manual intervention.

How it works: Kubernetes-based controller that reconciles MCPServer custom resources. When you deploy a new MCP server and create an MCPServer object pointing to its HTTPRoute, the Controller picks it up, extracts the tool prefix (e.g., jira/) and target reference, and reconfigures the Broker and Router on the fly.

How They Work Together

Let's walk through a single tool call from an AI agent:

  1. Agent sends request: POST /mcp with JSON-RPC payload {"method": "tools/call", "params": {"name": "jira/create_issue", ...}}
  2. Router intercepts (ext_proc): Envoy pauses, calls the Router component via gRPC, passes the request body
  3. Router parses & sets headers: Extracts tool name jira/create_issue, looks up backend, sets X-MCP-Target-Server: jira-backend
  4. Envoy routes to backend: Uses the header to match routing rules and forwards the request to the Jira MCP server pod
  5. Backend MCP server executes: Jira MCP server receives the validated request (after auth policies applied by RHCL/Authorino) and creates the issue
  6. Response flows back: Jira MCP server returns JSON-RPC response → Envoy → Router (optional response processing) → Agent

Meanwhile, the Controller is running in the background, watching for new MCPServer resources. When you deploy a new GitHub MCP server, the Controller automatically adds it to the Broker's backend list and teaches the Router to recognize github/* tool prefixes.

MCP Gateway in the RHCL Stack

MCP Gateway doesn't run alone — it sits within the Red Hat Connectivity Link (RHCL) platform, which provides the underlying networking, auth, and rate limiting infrastructure.

Traffic flow from top to bottom:

  1. Gateway API: Kubernetes standard for configuring ingress (replaces older Ingress API)
  2. Ingress Gateway (Envoy): Entry point for external traffic into the cluster
  3. Limitador: Rate limiting enforcement (e.g., "100 requests/second per user")
  4. Authorino: Authentication engine (validates OAuth tokens, enforces AuthPolicy rules)
  5. MCP Gateway: Parses MCP protocol, routes to correct backend, aggregates tool lists
  6. MCP Servers: Backend services that expose tools (Jira, GitHub, internal APIs)

Each layer adds a governance capability. By the time a request reaches the backend MCP server, it has already been:

  • Rate-limited (Limitador)
  • Authenticated (Authorino)
  • Routed to the correct backend (MCP Gateway Router)
  • Logged and metered (throughout the stack)

Custom Resource Definitions (CRDs)

MCP Gateway introduces two Kubernetes CRDs for declarative configuration:

MCPServer

Registers an MCP server with the gateway. Specifies:

  • toolPrefix: The tool namespace this server owns (e.g., jira/)
  • targetRef: Reference to the Kubernetes HTTPRoute that routes traffic to this backend

Example: An MCPServer for Jira would have toolPrefix: jira/ and targetRef pointing to a route that sends traffic to the Jira MCP server pods.

MCPGatewayExtension

Configures gateway-level behavior. This is where you specify global settings like:

  • Which Gateway resource to attach to
  • Broker configuration (aggregation rules, virtual server settings)
  • Router configuration (parsing rules, header mappings)

Typically one per MCP Gateway deployment.

For detailed CRD schemas and configuration examples, see Build It > CRDs & Configuration.

Key Architectural Insight

MCP Gateway is a WASM plugin, not a standalone product.

The gateway components (Broker, Router, Controller) run as a WASM (WebAssembly) plugin inside Envoy. This means:

  • Minimal disruption: Users already running Envoy for ingress (OpenShift Service Mesh, Istio, standalone Envoy) don't need to rip-and-replace infrastructure.
  • Lightweight: WASM is compiled to near-native performance, hardware-independent, and universally supported.
  • Portable: The same WASM binary runs across different Envoy-based environments without recompilation.

This design choice makes MCP Gateway compatible with the broader Kubernetes ecosystem and easy to adopt incrementally.
