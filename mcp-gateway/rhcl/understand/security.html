Security

MCP Gateway implements enterprise-grade authentication and authorization using OAuth 2.0, OpenID Connect, token exchange, and cryptographic delegation chains — enabling secure, auditable agent-to-tool interactions.

Security Foundations: Understanding Tokens

What is a token? Think of it like a movie ticket. When you buy a ticket at the box office, you get a piece of paper that:

  • Proves you paid (authentication)
  • Says which movie and showtime (authorization — this ticket is only valid for Screen 3 at 7pm)
  • Has an expiration (you can't use it next week)
  • Can't be easily forged (special paper, barcodes, holograms)

A security token works the same way in software. It's a digitally signed piece of data that proves:

  • Who you are (identity / authentication)
  • What you're allowed to do (permissions / authorization)
  • When it expires (validity period)
  • It hasn't been tampered with (cryptographic signature)

MCP Gateway uses tokens extensively — but not just any tokens. Let's build up the concepts step by step.

OAuth 2.0 and OpenID Connect (OIDC)

OAuth 2.0 is a protocol for "delegated authorization" — giving a third party limited access to your resources without sharing your password.

Real-world analogy: You give a valet driver your car keys. But instead of handing over your house keys too, you give them a special valet key that only starts the car and doesn't open the trunk or glove box. That's delegation — limited access for a specific purpose.

In OAuth terms:

  • You = Resource Owner (the human user)
  • Car = Protected Resource (e.g., your Jira account)
  • Valet = Client (the AI agent)
  • Valet key = Access Token (proof of limited permission)
  • Authorization server = The system that issues tokens (Keycloak, Okta, etc.)

OpenID Connect (OIDC) is an identity layer on top of OAuth 2.0. OAuth handles "what can I access," OIDC adds "who am I." It introduces the ID token (a token that proves your identity) alongside the access token (which proves permission).

Both OAuth and OIDC typically use JWT (JSON Web Tokens) as the token format. A JWT is a JSON object that's base64-encoded and cryptographically signed, so you can verify it came from the right issuer and hasn't been modified.

The Token Exchange Problem

Here's the core challenge for AI agents:

  1. A human user authenticates and gets Token 1 (their personal access token for the enterprise)
  2. They send a request to an AI agent
  3. The AI agent needs to call a backend tool (Jira, GitHub, etc.) on the user's behalf
  4. But Token 1 was issued for the human user, not the agent. The backend service expects a token specific to the tool's audience.

Solution: Token Exchange. The agent takes Token 1 and exchanges it for Token 2 (scoped for the specific backend service). This is a standardized OAuth 2.0 extension (RFC 8693).

Token Exchange happens at the gateway layer. The agent doesn't need to implement OAuth flows or credential management — it just passes the user's token to the gateway, and the gateway handles the exchange.

Security Architecture: Three-Tier Token Flow

Let's walk through a real MCP Gateway request with token exchange:

User
Token 1 (user identity)
AuthProxy
Exchanges to Token 2
MCP Gateway
Exchanges to Token 3
MCP Server
Validates Token 3

Step-by-step flow:

  1. User authenticates: Human user logs in to the enterprise IdP (Okta, Keycloak, Azure AD) and receives Token 1.
    • Token 1 contains: user identity, enterprise-wide permissions, issued by the IdP
  2. User calls agent: User sends a request to the AI agent, passing Token 1 in the Authorization header.
  3. Agent calls gateway: Agent forwards the request to MCP Gateway, passing Token 1.
  4. AuthProxy intercepts (Token Exchange #1):
    • An Envoy sidecar (AuthProxy, part of the Kagenti security stack) intercepts the request before it reaches the gateway.
    • It calls the authorization server: "User presented Token 1. I need a token for audience 'mcp-gateway'."
    • Authorization server returns Token 2 (scoped for the MCP Gateway audience, still represents the user).
    • AuthProxy replaces Token 1 with Token 2 in the Authorization header and forwards the request to the gateway.
  5. MCP Gateway routes & exchanges (Token Exchange #2):
    • Gateway Router parses the MCP request, identifies the target tool (e.g., jira/create_issue).
    • Gateway exchanges Token 2 for Token 3 (scoped for the Jira MCP server audience).
    • Gateway forwards the request to the Jira MCP server with Token 3.
  6. MCP Server validates and executes:
    • Jira MCP server receives Token 3.
    • Envoy sidecar (if present) validates Token 3's signature and expiry using local JWT verification — no round-trip to the auth server.
    • Jira MCP server is "dumb" — it just receives a validated token and executes the tool call. No auth logic required.

The "Dumb MCP Server" Pattern

From a customer security discussion (May 2026, confirmed by the Kagenti engineering team):

"AuthBridge is designed to transparently handle offloading the majority of the auth work so that the MCP server can be quite dumb. The MCP server receives a validated token in its Authorization header. No token exchange, no credential management, and no auth code is required in the MCP server itself. The Envoy sidecar (injected by Kagenti) handles JWT validation transparently."

Why this matters: MCP server developers don't need to become OAuth experts. They write tool logic (create Jira issue, query database, etc.) and assume the token is already valid. The gateway and sidecar infrastructure handle all the auth complexity.

This dramatically lowers the barrier to building secure MCP servers and reduces the attack surface (less auth code = fewer auth bugs).

Token Caching: Avoiding Redundant Exchanges

Token exchange is a network round-trip to the authorization server. Doing this on every request would be slow. MCP Gateway caches tokens to avoid redundant exchanges.

Implementation details (from Kagenti AuthBridge):

  • Cache location: In-process memory (thread-safe map within the AuthProxy/Gateway process)
  • Cache key: SHA-256 hash of (subjectToken, audience) — ensures the right cached token is retrieved for each unique user + target combination
  • Lazy eviction: Tokens are evicted when their TTL expires
  • Pre-expiry refresh: Tokens are refreshed 30 seconds before expiration to ensure they're never served stale
  • Max size: Default 10,000 entries (configurable). When full, least-recently-used entries are evicted.
  • External caching (Redis, etc.): Not currently supported but planned. For single-replica deployments, in-process memory is sufficient. Multi-replica gateways sharing a cache require external storage.

Result: After the first request from a user to a specific tool, subsequent requests hit the cache and skip the token exchange round-trip entirely — until the token expires or is evicted.

On-Behalf-Of (OBO) Delegation

What is On-Behalf-Of? It's the pattern where an agent acts with a subset of the user's permissions, not the agent's own permissions.

Real-world analogy: You give your assistant power of attorney to sign documents on your behalf — but only for specific types of documents (contracts, not wills). The assistant isn't signing as themselves; they're signing as you, but with limited scope.

In MCP Gateway:

  • Agent A is running on behalf of User X.
  • User X has broad permissions (admin access to Jira, GitHub, databases).
  • Agent A should only have a subset of those permissions (read-only Jira, no GitHub, no database admin).

OBO is implemented via token exchange with scoped audiences:

  • The agent requests a token for a specific tool/audience (e.g., "jira-read-only").
  • The authorization server (Keycloak) checks the user's permissions and the agent's allowed scopes.
  • It issues a token that represents the user but with limited permissions — the intersection of what the user is allowed and what the agent is allowed.

This prevents privilege escalation: even if a user has admin rights, the agent can only exercise the subset of rights it was configured to use.

Cryptographic Delegation Chains: The act Claim

How do you trace a tool call back to the human who initiated it when it passes through multiple agents and gateways?

Answer: The act (actor) claim in JWTs. This is a standardized claim (RFC 8693 §4.1) that chains delegation.

How it works:

  • Token 1 (user token) has sub: "user@example.com" (subject = the user)
  • Token 2 (exchanged for MCP Gateway) has sub: "user@example.com", act: {"sub": "mcp-gateway"} — meaning "the subject is the user, acting through the MCP Gateway"
  • Token 3 (exchanged for Jira MCP server) has sub: "user@example.com", act: {"sub": "mcp-gateway", "act": {"sub": "jira-mcp-server"}} — a nested chain

This creates a cryptographically signed trail:

  1. User authenticated as user@example.com
  2. Delegated to mcp-gateway
  3. Which delegated to jira-mcp-server

The backend MCP server can inspect this chain and know: "This request came from user@example.com, through the MCP Gateway, to me." Audit logs capture this chain for compliance.

Full Audit Trail

From a customer security discussion (May 2026):

  • Keycloak emits events for token exchange activities (user → agent, agent → backend)
  • Gateway logs every tool call with the act claim chain
  • MCP servers log tool execution with the validated token's subject and actor chain

Combined, this provides a complete trail:

  • Who: user@example.com
  • Delegated through: mcp-gateway
  • Called what: jira/create_issue
  • With what permissions: jira-read-only scope
  • When: Timestamp in logs
  • Result: Success/failure in MCP server logs

This is enough to satisfy SOC 2, ISO 27001, and financial services audit requirements for "who did what and when."

Multi-IdP Support: Okta, Azure AD, Keycloak

MCP Gateway doesn't dictate which identity provider you use. It works with any OIDC-compliant IdP:

  • Okta: Federate via OIDC identity brokering in Keycloak. Use protocol mappers to map group claims.
  • Azure AD / Entra ID: Same — OIDC federation + claim mapping.
  • Keycloak: Native integration (Keycloak is the default authorization server in the Kagenti stack).
  • AWS Cognito, Auth0, etc.: Any OIDC-compliant IdP works.

Nested groups: Traditionally flattened when presented (e.g., "Engineering/Platform/MCP" becomes three separate group memberships). Reconstructing hierarchies requires custom mappers.

Sync timing: Configurable per IdP — sync at first login only, or sync every time the user authenticates (to pick up group membership changes).

Security Best Practices

Recommended Patterns
  • Short-lived tokens: Set token TTL to 5-15 minutes. The cache handles renewal, so short TTLs reduce risk without hurting performance.
  • Principle of least privilege: Configure OBO scopes so agents only get the minimum permissions needed for their tasks.
  • Audit log retention: Keep Keycloak event logs and gateway logs for at least 90 days (longer for compliance-heavy industries).
  • Token rotation: Use refresh tokens for long-running sessions; re-exchange when access tokens expire.
  • Network isolation: MCP servers should only accept traffic from the gateway (use network policies or service mesh rules).

What About Secrets Management?

Token exchange solves authentication between the user, gateway, and MCP servers. But what if the MCP server needs to authenticate to a backend service that doesn't support OAuth (e.g., a database with username/password, GitHub Personal Access Tokens)?

Answer: The MCP server stores those credentials securely (e.g., Kubernetes Secrets, Vault, AWS Secrets Manager). The MCP server retrieves them at runtime and uses them to authenticate to the backend.

RHCL 1.3.3 includes Vault integration (Technical Preview), allowing MCP servers to dynamically fetch secrets without storing them in container environment variables.
