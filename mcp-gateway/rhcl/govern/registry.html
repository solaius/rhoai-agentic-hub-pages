MCP Registry

The authoritative governance layer for MCP servers in RHOAI. System of record that brings governance, lifecycle control, trust, and visibility so MCP assets can move cleanly from discovery through approval, deployment, runtime use, and consumption.

Target: RHOAI 3.5 Dev Preview

Purpose

System of record for governed MCP servers (internal or external). Provides:

  • Ingestion and lifecycle management
  • Security and certification context
  • Versioning and immutability
  • Policy-aware visibility

MCPServer

Governance Model

The current data model proposal has collapsed governance into two independent enums, not the four-track model described in earlier drafts:

Enum Scope States
MCPServerStatus Asset-level (MCPServer) Active → Deprecated → Retired
MCPPublishState Version-level (MCPServerVersion) Draft → Published → Deprecated → Retired
Not yet modeled: The current dataclass has no explicit Approval Status, Verification Status, or Certification Status fields. Certification tracking remains an open question, not a modeled track.
Design Principle: Governance metadata is first-class and separate from tags. The server_json on MCPServerVersion stores the canonical MCP payload following the upstream MCP registry schema.

Observed Tool Metadata

Two new entities separate cached, non-authoritative tool metadata from the canonical registry payload:

  • MCPObservedTool: a tool as observed from a live MCP endpoint, tracked independently of the server's canonical server_json.
  • MCPObservedToolSnapshot: a point-in-time snapshot of observed tool metadata, intended for display and search UX.

Populated either manually or by syncing the MCP client's list_tools() output.

MLflow Integration

Red Hat is collaborating upstream with MLflow/Databricks on MCP registry design. The data model proposal is undergoing RFC approval process. This collaboration informs the future generalized AI asset registry architecture.

MVP Scope (3.5 Dev Preview)

  • Governed registration of MCP servers
  • Metadata-first MCP records
  • Lifecycle state tracking
  • Certification and approval visibility
  • RBAC-aware visibility and governance
  • Support for internal and external MCP representation
  • Conceptual integration with catalog, deployment, gateway, Studio

Open Questions (from Reviewers)

  • Dan Kuc: MLflow lifecycle states (None/Staging/Production/Archived) may need expansion for MCP
  • Jon Burdo: Certification allowlist question may connect to RHAIRFE-294
Resolved (managed registry entries): Chris Hambridge's question ("should Lifecycle Operator create read-only 'managed' registry entries?") is answered. An update_mode field (MANUAL vs SYSTEM_MANAGED) governs whether a registry entry's mutable fields are user-edited or kept in sync by another system. This is the concrete mechanism behind Matt Prahl's "discovered" MCP server concept: system-populated entries where only description and tags remain user-editable.
Integration Gap: Not yet integrated with Gateway — automated registry→gateway flow is GA target (Q4 2026). Today's manual gateway registration will be replaced by registry-driven configuration.
