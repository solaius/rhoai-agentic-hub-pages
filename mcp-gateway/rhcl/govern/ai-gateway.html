AI Gateway: The Bigger Picture

The MCP Gateway handles MCP protocol traffic. The AI Gateway covers ALL AI traffic.

Source: AI Gateway F2F (April 2026) — binding architectural decisions from 3-day engineering F2F.
Status: Architecture decided (~60% detailed); follow-up design sessions needed before 3.5 scope is fully locked.

Four Traffic Types

The AI Gateway must serve four distinct traffic types:

# Type Examples
1 Model inference /v1/chat/completions, /v1/embeddings
2 Tool calling / MCP MCP Gateway handles this
3 Agent-to-Agent (A2A) AgentCard discovery, invocation, orchestration
4 External provider egress Route to Azure OpenAI, Bedrock, Vertex with credential injection

IPP Framework

The Inference Payload Processor (IPP) is the foundational plugin layer for ALL gateway processing — inference, egress, guardrails, metering, and the Responses API.

  • Implemented as ext_proc plugin chain within Envoy
  • Current plugin pipeline: body-field-to-header → model-provider-resolver → api-translation → apikey-injection
  • Policy flows currently in Authorino (separate ext_proc) will migrate into IPP plugins within a single ext_proc, reducing architectural complexity
  • Designed to be proxy-agnostic — architecture is sound regardless of underlying proxy (Envoy, Praxis, etc.)
Relationship: MCP Gateway and AI Gateway are complementary, not competing. Both share Envoy, Gateway API, and RHCL infrastructure (Authorino, Limitador).

Shared Infrastructure

Both MCP Gateway and AI Gateway rely on:

  • Envoy Proxy: Underlying proxy infrastructure
  • Gateway API: Kubernetes Gateway API for routing
  • RHCL Platform: Red Hat Connectivity Link (Authorino, Limitador, DNS Operator)
  • ext_proc Plugins: Extension processor plugin chains

Key F2F Decisions Affecting MCP Gateway

1. Group-Based Tenancy

The existing group-based subscription model for models in MaaS is generalized to:

  • Vector stores
  • File storage
  • MCP tool catalogs
  • Guardrails
  • Evaluations

2. Agent Identity Tiers

Tier Mechanism What It Provides
Tier 1 OpenShell token masking Prevents agents from seeing raw user credentials; simple to adopt
Tier 2 Spire/Authbridge integration Full workload identity with actor-chain tracking; OPA policy enforcement

Actor-chain tracking: When agent A calls agent B, all actors' SPIFFE IDs stored in the token's claims. OPA enforces the lowest permission level across the chain — prevents privilege escalation.

3. Guardrails Convergence

MCP Gateway already ships guardrails (NeMo/TrustyAI) via ext_proc adapter. The same guardrails are needed for chat completions, Responses API, and external model egress. Convergence target: Extract guardrails into shared IPP plugin for both inference and MCP.
