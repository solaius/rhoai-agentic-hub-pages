Resolved Questions

Questions that have been answered or closed. For reference only.

Okta Group Mapping in Keycloak

Question: How do Okta groups map into Keycloak for authorization decisions? (Customer Q1)

Answer: Use OIDC federation (not LDAP sync). Keycloak identity-brokers with Okta via OIDC. Use protocol mappers to map group claims from the source Okta token to the Keycloak-minted token. Nested groups are traditionally flattened; reconstructing a hierarchy is possible but requires custom work. Sync timing is configurable: sync at first login only, or sync every time the user authenticates.

Status: Resolved (confirmed by the Kagenti engineering team, 2026-05-06)

External Keycloak Support

Question: Can you use any Keycloak instance, or does it need to be on the same cluster/namespace? (Customer Q3)

Answer: On the MCP Gateway side, Keycloak can be hosted anywhere — no co-location requirement. On the Kagenti side, tested with in-cluster Keycloak, but it should work with any external Keycloak as long as it is exposed. No shared SPIFFE trust domain is needed — SPIFFE is used as a client ID.

Status: Resolved (confirmed and updated by the Kagenti engineering team, May 2026)

Token Caching & Consecutive Calls

Question: How does AuthBridge handle consecutive calls? Does it re-exchange every time? (Customer Q6, Q7)

Answer: No. AuthBridge's in-process cache stores exchanged tokens keyed by (subjectToken, audience) and serves them until TTL expiry (with a 30-second early-refresh buffer). Consecutive calls do not re-exchange. External cache (Redis) is not yet supported but Kagenti team is willing to add it.

Status: Resolved (confirmed via code review + Kagenti engineering team, May 2026)
