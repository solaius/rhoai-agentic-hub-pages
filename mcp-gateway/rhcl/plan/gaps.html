Known Gaps

Technical gaps identified through customer POCs, partner evaluations, and internal testing.

Critical Gaps

These gaps represent missing features that are expected for production use. All are planned for GA (Q4 2026).

Gap Health Checks & Circuit Breakers

What's Missing: Automated health checking and circuit breaker logic to detect and remove unhealthy MCP servers from rotation.

Current State: Not implemented. Envoy provides basic upstream health checking at the infrastructure layer, which can be used as an interim solution.

Roadmap: Planned under "High Availability" feature set for GA (Q4 2026). Will include backend health checks and automatic failover.

Impact: Table-stakes for production. Manual intervention required to remove unhealthy backends today.

Source: Partner POC criteria #5

Gap Registry → Gateway Auto-Configuration

What's Missing: Automated flow from MCP Registry approval to Gateway configuration. Today, registry and gateway are separate manual processes.

Current State: Manual configuration via CRDs and ConfigMaps. When an MCP server is approved in the registry, a cluster admin must separately create an MCPServer resource and configure routing.

Roadmap: Registry integration is a GA milestone (Q4 2026). Target: registry approval → gateway configuration flows automatically.

Impact: Operational friction. Increases time-to-production and creates opportunity for configuration drift.

Source: Partner POC criteria #1, knowledge-review

Partial Gaps

These features are partially implemented. Core functionality exists but some capabilities are missing.

Partial Per-Tool QoS (Timeouts/Concurrency)

What Works: Rate limiting per tool via Limitador (RHCL component). Limits can be configured per agent, per tool, or globally.

What's Missing: Timeouts and concurrency limits are not yet configurable per tool through MCP Gateway CRDs.

Workaround: Envoy provides infrastructure-level timeout configuration that can be used until CRD-level controls are available.

Roadmap: Future enhancement (timeline TBD).

Source: Partner POC criteria #6

Partial WebSocket & Stdio Transport

What Works: HTTP and SSE (Server-Sent Events) streaming are fully supported at the gateway edge through the single /mcp endpoint.

What's Missing: WebSocket and stdio transports are not supported at the gateway edge.

Workaround: Client-side adapters or backend-side translation can bridge WebSocket/stdio MCP servers to HTTP/SSE at the gateway boundary.

Impact: Depends on whether backend MCP servers use WebSocket or stdio natively. If so, additional adapter infrastructure is required.

Source: Partner POC criteria #3

Partial Cost Tracking

What Works: Token metrics exist (tokens consumed per tool invocation, per agent, per time window). Per-tool invocation metrics, OpenTelemetry traces, and structured audit logging are all implemented.

What's Missing: Cost tracking is not explicitly modeled in metrics. There is no built-in mapping from tokens to cost.

Workaround: Token metrics can be exported and combined with external pricing data to calculate cost in downstream observability pipelines.

Roadmap: Cost tracking could be added as a metric dimension in future releases.

Source: Partner POC criteria #9

Partial Version-Aware Routing

What Works: MCP Registry tracks full lifecycle states (Draft → Candidate → Published → Deprecated → Retired) with immutable MCPServerVersion entities, approval workflows, and verification/certification status.

What's Missing: The Gateway does not enforce version-aware routing. There is no version pinning per agent or deprecation enforcement at the routing layer.

Impact: Agents always route to the latest available version. No ability to pin an agent to a specific MCP server version or to gracefully migrate between versions.

Roadmap: GA integration milestone (Q4 2026). Dependent on registry-to-gateway auto-configuration.

Source: Partner POC criteria #10

Partial External Cache (Redis)

What Works: AuthBridge includes in-process token caching. Exchanged tokens are cached in memory (SHA-256 keyed by (subjectToken, audience)) with lazy TTL eviction and 30-second pre-expiry refresh.

What's Missing: No external cache backend (e.g., Redis) for sharing cached tokens across multiple gateway replicas.

Workaround: For single-replica deployments, in-process memory is sufficient. For multi-replica HA deployments, each replica performs independent token exchanges (higher load on Keycloak).

Roadmap: The Kagenti team has confirmed willingness to add Redis support. The refactor is small: extract a Cacher interface and implement a Redis adapter.

Source: OEM customer discussion Q6

Partial OCP Version Compatibility

Issue: RHOAI 3.5 (Aug 2026) will support OCP 4.19, but RHCL 1.4 may not. This configuration (RHOAI 3.5 + RHCL + OCP 4.19) is untested and best-effort support only.

Context: OCP 4.19 maintenance support ends Dec 2026 — only 4 months after RHOAI 3.5 ships. Gateway API is GA in OCP 4.19+, but RHCL testing has focused on newer OCP versions.

Recommendation: Customers deploying RHOAI 3.5 with RHCL should plan to upgrade to OCP 4.20+ for full support.

Source: Support Agreement, knowledge-review

Partial TLS Issue in RHCL 1.4

Issue: GitHub issue #1130 — TLS hairpin and per-server SNI fixes were needed after RHCL 1.4 GA.

Status: Fixed in MCP Gateway v0.7.1 (Jun 18, 2026). RC2 patch in progress with expanded TLS matrix testing.

Impact: Resolved. Customers using RHCL 1.4 should upgrade to MCP Gateway v0.7.1.

Source: GitHub, Slack

Gap Analysis Summary

Of the partner's 10 POC success criteria: 4 fully met, 4 partially met, 2 gaps.

Strengths: Auth (#7, #8) and policy routing (#4) are differentiated capabilities that solve hard enterprise problems.

Weaknesses: Operational maturity (health checks, QoS completeness) and integration plumbing (registry→gateway auto-config).
