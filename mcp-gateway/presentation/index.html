Core Concept

A Runtime Control Plane for MCP Traffic

The MCP Gateway sits between your AI agents and the MCP servers they consume. It provides a single governed endpoint that handles authentication, authorization, routing, and observability — without changing a line of agent code.
Router
ext_proc
Parses the MCP protocol and routes requests to the correct backend server
Broker
aggregator
Aggregates tools from registered MCP servers into virtual endpoints
Controller
k8s operator
Watches Kubernetes resources and auto-discovers new MCP servers
Emphasize zero agent code changes. The WASM plugin architecture means it's lightweight and non-disruptive to existing Envoy/Istio infrastructure. The three components work together: Controller discovers, Broker aggregates, Router directs.

Explain the RHCL relationship. Gateway API (not Ingress) is the routing standard. Authorino handles auth, Limitador handles rate limiting. The gateway is a WASM plugin that runs inside Envoy, not a separate process.

Call out which capabilities come with RHOAI vs which require RHCL upgrade. Virtual MCPs and auth are RHOAI-included; rate limiting requires RHCL. The four categories map to different buyer concerns: ops, security, SRE, and governance.
