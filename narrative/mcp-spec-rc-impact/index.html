MCP PROTOCOL — RELEASE CANDIDATE

2026-07-28 RC Impact on RHOAI

The largest revision of the Model Context Protocol since launch eliminates protocol-level sessions, removes the initialize handshake, redesigns the Tasks API, and hardens authorization. This analysis maps every change against RHOAI 3.4 / 3.5 architecture.

WHAT CHANGED

Spec Overview

Six SEPs collectively eliminate protocol-level statefulness. Combined with auth hardening, a redesigned Tasks API, and a new extensions framework, this is a fundamental shift in how MCP works at the wire level.

ARCHITECTURE SHIFT

The Stateless Protocol Core

The single biggest change in this RC. Six SEPs collectively eliminate protocol-level statefulness, transforming how every component in the RHOAI stack interacts with MCP servers.

Before (2025-11-25)
// Connection setup required POST /mcp "method": "initialize" "params": { clientInfo, capabilities } // Server responds with session Mcp-Session-Id: abc-123 // All subsequent requests pinned POST /mcp Mcp-Session-Id: abc-123 "method": "tools/call"
  • Sticky routing required
  • Shared session stores for HA
  • Body inspection for routing
  • Single-instance affinity
After (2026-07-28)
// No handshake — direct call POST /mcp MCP-Protocol-Version: 2026-07-28 Mcp-Method: tools/call Mcp-Name: search Content-Type: application/json { "_meta": { "clientInfo": { "name": "rhoai" } }, "method": "tools/call" }
  • Any request hits any instance
  • Header-based routing (no body parse)
  • Standard K8s load balancing works
  • No session stores needed

Key Mechanisms

Explicit Handle Pattern

Servers that need cross-call state mint explicit handles (e.g., basket_id, browser_id). The model threads these as ordinary tool arguments. State is visible to the model, not hidden in protocol-level sessions.

Capabilities via server/discover

Client capabilities travel in _meta on every request. Server capabilities are fetched on-demand via server/discover and cached using ttlMs/cacheScope fields.

Multi Round-Trip Requests SEP-2322

Replaces holding SSE streams open. Server returns InputRequiredResult with requestState token. Client gathers input, re-issues the call with the token. Any server instance can handle the retry.

Cache-Control Model SEP-2549

List and read results carry ttlMs (freshness) and cacheScope (shareability). Eliminates need for long-lived SSE streams to detect tools/list changes. Critical for multi-tenant deployments.

BREAKING CHANGES

7 Breaking Changes

Every breaking change in the RC, with the protocol-level detail engineers need to assess implementation impact.

# Change SEP Detail
1 Initialize handshake removed SEP-2575 initialize/initialized methods deleted. Client info and capabilities now in _meta per-request. server/discover replaces capability negotiation.
2 Sessions eliminated SEP-2567 Mcp-Session-Id header removed. Protocol-level session concept gone. No sticky routing, no shared session stores at protocol layer.
3 Required routing headers SEP-2243 Mcp-Method and Mcp-Name required on Streamable HTTP. Servers must reject requests where headers and body disagree.
4 Tasks API redesigned SEP-2663 Moved from core to extension. tasks/list removed. Creation is server-directed. Client drives via tasks/get/tasks/update/tasks/cancel with task handles.
5 Error code changed SEP-2164 Missing resource error code: -32002 (MCP-custom) → -32602 (JSON-RPC standard Invalid Params).
6 Per-request client info SEP-2575 Client info and capabilities must be sent in _meta on every request, not negotiated once at connection time.
7 Server request scoping SEP-2260 Server-initiated requests may only be issued while actively processing a client request. Was recommendation, now required.
CRITICAL COMPONENT IMPACT

MCP Gateway

The most impacted component. The Gateway's routing, session management, and auth layers are all directly affected. However, many changes are net-positive for the Envoy-based architecture.

github.com/Kuadrant/mcp-gateway ↗
Critical Breaking
Session Elimination → Routing Model
  • Gateway currently uses Mcp-Session-Id for sticky routing — header is removed
  • ext_proc component's session-based routing logic must be replaced
  • Shared session stores become unnecessary at protocol layer
  • "Session Management" planned for GA (Q4 2026) needs redefinition as application-level, not protocol-level
Critical Breaking
Initialize Removal → Broker
  • Broker handles initialize/initialized as part of connection setup — deleted
  • Must parse _meta per-request instead of caching from initialization
  • Capability negotiation cache invalidated — use ttlMs-based caching
High Positive
Required Headers → Router (ext_proc)
  • Mcp-Method + Mcp-Name enable header-only routing — no JSON-RPC body parsing needed
  • Perfect fit for Envoy ext_proc — reduces latency, simplifies Router
  • Rate-limiting and access policies can now key on headers directly
  • Must validate header-body agreement and reject mismatches
High
Auth Hardening → Authorino Integration
  • RFC 9207 iss validation required on auth responses
  • Credential binding to issuing authorization server's issuer
  • OIDC application_type in Dynamic Client Registration
  • Authorino handles much of this, but Gateway's MCP-specific auth layer needs updating
Medium Positive
Caching → Cache Component
  • ttlMs enables smart caching without long-lived SSE streams
  • cacheScope indicates per-user vs shared — critical for multi-tenant RHOAI
  • Gateway Cache component can implement native TTL-based invalidation
Medium Positive
W3C Trace Context → OTEL
  • Standardized traceparent/tracestate/baggage in _meta
  • Gateway OTEL tracing (planned TP) now has a standard propagation format
  • End-to-end traces: Studio → Gateway → MCP Server → downstream

Net assessment: The stateless model is architecturally better for the Gateway. Eliminates session stores, enables standard K8s load balancing, and header-based routing simplifies the ext_proc component. But the rework is significant — session routing, initialize handling, and auth layers all need updates before GA.

HIGH COMPONENT IMPACT

OGX / Llama Stack Operator

The Tasks API redesign is a breaking change for agent workflows running on Llama Stack distributions. OGX must adopt the extension-based Tasks API or lose long-running task support.

github.com/red-hat-data-services/ogx-k8s-operator ↗   github.com/meta-llama/llama-stack ↗
High Breaking
Tasks API Redesign SEP-2663
  • Tasks moved from core experimental to extension
  • tasks/list removed entirely — cannot be scoped without sessions
  • Task creation is now server-directed: client advertises extension, server decides when to use
  • Client drives lifecycle: tasks/get, tasks/update, tasks/cancel
  • Any OGX agent distribution using the 2025-11-25 experimental API will break
Medium
Extension Negotiation
  • OGX operator must advertise Tasks extension support in per-request _meta
  • Extensions negotiated via extensions map on capabilities
  • Llama Stack distributions may need updates to support the new extension model
  • Distribution CRDs (starter, remote-vllm, etc.) unaffected — these are deployment config, not protocol

Tasks Lifecycle: Before & After

2025-11-25 (Experimental)
// Core protocol, session-scoped tasks/create → server creates task tasks/list → list all tasks in session tasks/get → check status tasks/cancel → abort
2026-07-28 (Extension)
// Extension, stateless, handle-based tools/call → server returns task handle tasks/get → poll status via handle tasks/update → send input to task tasks/cancel → abort via handle // tasks/list REMOVED

Action required: Assess whether Llama Stack distributions in RHOAI 3.4 use the experimental Tasks API for agent workflows. If yes, this is a blocker requiring coordination with Meta/Llama Stack upstream.

HIGH COMPONENT IMPACT

MCP Registry

Registry governance metadata is independent of the MCP spec (good), but the canonical server_json payload, extension tracking, and JSON Schema storage all need updates.

github.com/kubeflow/model-registry (renaming to kubeflow/hub) ↗
High
server_json Payload Format
  • Canonical MCP payload stored per registry entry needs updating for new protocol version
  • New MCP-Protocol-Version: 2026-07-28 header format
  • _meta structure changes (client info, capabilities)
  • Handle-based state model changes how Registry tracks active server state
Medium
Extension Tracking
  • Registry must track which extensions an MCP server supports as governance metadata
  • Extensions use reverse-DNS IDs, version independently of core spec
  • Tasks extension support becomes a registerable capability
  • MCP Apps extension (interactive HTML UIs) has security review implications
Medium
JSON Schema 2020-12 SEP-2106
  • Tool inputSchema/outputSchema now support oneOf, anyOf, allOf, $ref, $defs
  • Registry's tool metadata storage must handle richer schemas
  • Must NOT auto-dereference external $ref URIs (security)
  • Should bound schema depth and validation time
Low Positive
Governance Independence
  • Four governance tracks (Lifecycle, Approval, Verification, Certification) are RHOAI-defined, not spec-defined
  • Lifecycle states, audit trail, mandatory comments — unaffected
  • Registry's value proposition (governance gap) is strengthened by spec going stateless

STRATEGIC ASSESSMENT

Impact Summary

Component Severity Key Changes
MCP Gateway Critical Session elimination, header-based routing, auth hardening, caching, trace context, stateless routing
OGX / Llama Stack High Tasks API completely redesigned, new extension model for long-running workflows
MCP Registry High server_json format update, extension tracking, richer JSON Schema, handle-based state
MCP Catalog Medium Caching TTLs, extension awareness, richer schema rendering, auth changes
Lifecycle Operator Medium Health probes (no more initialize), stateless deployment becomes simpler
Studio / AAA Medium OIDC app_type for DCR, multi round-trip UX, MCP Apps rendering
What's Good
  • Stateless = K8s-native. Eliminates sticky routing, shared session stores, and single-instance affinity. Standard Kubernetes load balancing just works.
  • Header-based routing. Perfect for Envoy ext_proc. Reduces Gateway latency by eliminating JSON-RPC body parsing for routing decisions.
  • Native caching. ttlMs/cacheScope gives Gateway and Catalog smart caching without long-lived SSE streams.
  • OTEL alignment. W3C Trace Context propagation aligns with RHOAI's existing observability stack.
  • Simpler HA. No session stores means fewer moving parts for high availability.
  • Logging deprecation. Move to OTEL aligns with what RHOAI already uses.
Concerns
  • Gateway rework scope. Session routing, initialize handling, and auth layers all need updates before GA. Significant implementation effort.
  • 10-week RC window. May 21 → July 28 is tight for the scope of Gateway changes. SDK tier system expects Tier 1 support in this window.
  • OGX Tasks API. Breaking change for agent workflows. Need to assess Llama Stack upstream timeline for extension adoption.
  • Gateway session management. Planned GA feature needs complete redefinition — what does "session management" mean in a stateless protocol?
  • MCP Apps security. Interactive HTML UIs in sandboxed iframes has significant security review implications for RHOAI's compliance posture.
  • Auth migration. 6 auth SEPs require coordination between Gateway, Authorino, and Catalog teams.

Recommended Actions

1
Flag to Kuadrant/Gateway team immediately

Session elimination and routing changes are architecturally positive but require significant rework. Needs planning within the 10-week RC window.

2
Assess OGX Tasks API usage

Determine if Llama Stack distributions in RHOAI 3.4 use the experimental Tasks API. Coordinate with Meta upstream if so.

3
Update MCP Registry server_json schema

Add protocol version tracking, extension metadata, and JSON Schema 2020-12 support to the Registry data model before 3.5 Dev Preview.

4
Coordinate auth hardening across teams

6 auth SEPs affect Gateway, Catalog, and Studio. RFC 9207 issuer validation and credential binding need coordinated implementation.

5
Redefine Gateway "Session Management" feature

Protocol-level sessions no longer exist. Redefine as application-level state management with explicit handles, or descope for GA.
