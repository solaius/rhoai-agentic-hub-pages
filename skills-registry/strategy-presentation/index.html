THE PROBLEM

Skills Are Proliferating Without Governance

AI skills are spreading across frameworks, formats, and distribution channels. Enterprises have no visibility into what's running, who approved it, or what version is in use.

Supply Chain Attack

The ClawHub crisis revealed 12-20% of skills in open registries were malicious — including CVE-2026-25253 enabling remote code execution. Skills face the same supply chain risks as npm/PyPI, plus prompt injection vectors.

Format Fragmentation

Skills exist as SKILL.md files, Python packages, npm modules, OCI containers, zip bundles, OpenAPI specs, and more. No consistent metadata, no interoperable manifest format, no standard lifecycle.

No Governance Layer

When an engineer grabs a skill from GitHub, there is no versioning, no approval workflow, no deprecation process, no dependency tracking. Once consumed, execution is invisible to the platform.

"There is no standardized skills registry anywhere. The market has tool definitions, agent protocols, and community catalogs — but no one has built the governance layer."

TAXONOMY

What Is a Skill?

A named, versioned, reusable AI capability that may be backed by MCP servers, CLI tools, prompt templates, instruction sets, or code libraries. Skills sit above individual tools but below full agents.

Composed runtime capability — orchestrates skills to achieve goals

SKILL.md — The Emerging Standard

Launched by Anthropic (Dec 2025) and adopted by OpenAI Codex, Google Gemini CLI, GitHub Copilot, and VS Code. A directory with YAML frontmatter + markdown instructions + optional scripts/resources. Progressive disclosure: metadata at startup, instructions on demand, resources during execution.

Two Consumption Models

Client-Side (Dominant Today)

Agent downloads the skill artifact and runs it locally. Skill code executes in the agent's context. Registry enforces policy at resolution time, but post-download execution is outside registry control.

Server-Side (Emerging)

Server executes skills on behalf of agents via API (e.g., Responses API). Enables runtime controls, observability, and policy enforcement. Requires additional infrastructure.

Registry vs. Catalog

Registry (MLflow)
= Governance

System of record for identity, versioning, lifecycle, trust. "Who is this skill, what state is it in, is it approved?"

Catalog (Kubeflow Hub)
= Discovery

Browsing, searching, filtering, trust-tier presentation. "What skills are available to me?"

ARCHITECTURE

Data Model & Governance

The skills registry extends the MCP registry pattern: same backend, same governance tracks, same workspace scoping. The key difference — skills are consumed as artifacts, not deployed as services.

Pattern MCP Registry Skills Registry
Backend MLflow MLflow
Entity model MCPServer + MCPServerVersion Skill + SkillVersion
Governance tracks Lifecycle, Approval, Verification, Certification Same
Payload Immutable server_json Immutable skill artifact reference
Catalog Kubeflow Hub Kubeflow Hub
Deployment Lifecycle Operator + Gateway Not applicable — artifact consumption
