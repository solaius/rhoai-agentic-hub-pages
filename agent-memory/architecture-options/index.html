RHAISTRAT-1345 — AGENT MEMORY

Agent Memory Architecture Options

A composable architecture for enterprise agent memory in RHOAI. Three layers, multiple candidates per layer, and a phased roadmap from what we ship today to a complete governed solution.

ARCHITECTURE

Three Layers, Not One Monolith

Agent memory is not a single system. It decomposes into three distinct layers with different responsibilities, different maturity requirements, and different candidate projects for each.

LAYER 1

Memory Substrate

The raw storage and retrieval engine. Handles read path (vector search, context injection before inference) and write path (async summarization, vector store upload after response). Owns the Responses API integration point.

Vector search Context injection Async write Compaction
▼
LAYER 2

Memory Intelligence

The application-layer brain. LLM-based extraction (turning conversations into durable facts), hybrid retrieval (vector + BM25 + entity graph), contradiction detection, semantic deduplication, and PII scanning.

LLM extraction Contradiction detection Hybrid retrieval PII scanning
▼
LAYER 3

Governance & Scope

Enterprise policy enforcement. Multi-tier scope isolation (user/project/org), RBAC, immutable audit trails, erasure/retention, provenance tracking, and compliance (EU AI Act, GDPR, HIPAA). This is what turns a memory store into a governed memory store.

Scope tiers Audit trail RBAC Erasure Provenance

Emerging consensus (Jun 23 team sync): Long-term memory should be separate from the session/Responses API, but “they come together at prompt assembly time.” Memory as an independent, decoupled service — callable via MCP from any agent framework — would survive the OGX deprecation. All architectural paths below are proposals under evaluation for the July architecture review.

OPTIONS ANALYSIS

Candidates by Layer

Each layer has different candidates with different strengths. No single project fills all three layers well.

LAYER 1

Memory Substrate Options

OGX / AI Gateway Responses API IN RHOAI 3.5

Author: Francisco Arceo PRs #6162 / #6163 Community: Meta + Red Hat contributors
  • Already in the RHOAI 3.5 production path
  • Conversations, Vector Stores, Files, Prompts, Compaction all ship
  • Multi-tenancy (AuthorizedSqlStore ABAC) landed
  • Francisco is upstream maintainer — Red Hat has influence
  • OGX is being deprecated — AI Gateway absorbing its surfaces
  • Memory Alpha MVP tightly couples memory to Responses API lifecycle
  • No governance primitives (audit, scope tiers, provenance)
  • Owner-scoped only — no project/org tiers

PostgreSQL + pgvector (standalone) DECOUPLED

Used by: MemoryHub, Oracle AI Agent Memory Community: massive (PostgreSQL ecosystem)
  • Competitive with dedicated vector DBs under 50M vectors
  • Single-database design (relational + vector + graph in one engine)
  • OpenShift OOTB PostgreSQL operator available
  • FIPS inheritable from cluster
  • Requires building the memory service layer on top
  • No agent memory-specific tooling out of the box
  • No dedicated vector DB has Red Hat OCP Operator Certification
LAYER 2

Memory Intelligence Options

Mem0 EXTERNAL OSS

License: Apache 2.0 (OSS tier) GitHub: 48,000+ stars Funding: $24M Series A Community: large, fast-moving
  • Most popular open-source memory framework
  • Single-pass LLM extraction, hybrid retrieval (vector + BM25 + entity graph)
  • 21 framework integrations, 20 vector store backends
  • AWS Agent SDK exclusive memory provider
  • 93.4% on LongMemEval (vendor-published)
  • Governance thin in OSS tier (no audit, no erasure)
  • Drops to 48.6% at 10M-token enterprise scale (BEAM)
  • Open-core model — key features may migrate to paid tier
  • Market consolidation risk (5+ funded startups converging)
  • Benchmarks are vendor-published, not independently validated

MemoryHub (curation engine) RED HAT

License: Apache 2.0 Author: Wes Jackson (Red Hat SSA) Community: 1 author (IP transferred to team)
  • Inline curation pipeline: regex PII scan + embedding similarity dedup
  • Contradiction detection with threshold-based surfacing
  • Two-vector retrieval via RRF (query vector + session focus)
  • Covers both intelligence and governance in one package
  • Single-author project — bus factor of 1
  • No independent benchmarks on extraction quality
  • Extraction less sophisticated than Mem0's single-pass algorithm
LAYER 3

Governance & Scope Options

Feast GA IN RHOAI 3.4

License: Apache 2.0 Team: Umberto, Aniket, Nikhil, Amita + Jonathan (PM) Community: active upstream, Red Hat-owned
Phase A — 3.5
Discovery Loop
Feast MCP server for agent-assisted feature engineering. Agents discover data schemas safely via MCP.
Phase B — 3.5
Retrieval Loop
MemoryFeatureView with L0 (Redis KV) / L1 (Parquet summaries) / L2 (pgvector embeddings). feast init -t agent scaffolding.
Phase C — 3.6
Write Loop
Feast as pluggable backend for Mem0/Zep. “Mem0 owns the cognitive logic; Feast owns the data boundary.”
Source: Umberto Manganiello, “Unified Platform Agentic Memory Infrastructure” (May 2026)
  • GA in RHOAI 3.4 — ships today, OLM-managed
  • OIDC group-based + namespace-based RBAC
  • Point-in-time correctness (unique: reconstruct state at any timestamp)
  • Lineage graph (source → transformation → serving)
  • TTL declarative expiration, cell-level masking
  • MCP server (RHAIRFE-1916, 3.5 candidate)
  • MemoryFeatureView primitive designed (L0/L1/L2 tiered memory)
  • Dedicated team with upstream community
  • 3-phase roadmap already designed (A → B → C)
  • RBAC is per-feature-view, not per-record — Delta 1 needed
  • No write-event audit log — Delta 2 needed
  • No per-record provenance fields — Delta 3 needed
  • No contradiction detection, PII scanning, or hybrid retrieval
  • Feature store, not a memory service — memory-specific features all need building
  • Phase C depends on Mem0 supporting pluggable storage backends

MemoryHub (governance layer) RED HAT PROTOTYPE

License: Apache 2.0 Author: Wes Jackson — IP transferred to team Community: 1 author + Sanjeev evaluating
  • 6-tier scope model (user/project/campaign/role/org/enterprise)
  • Contradiction detection + inline curation (PII + dedup)
  • Memory versioning with full history + relationship edges
  • MCP server (3 profiles: compact/full/minimal)
  • UBI9 images, air-gap deployable, FIPS-delegated
  • SDK on PyPI with kagenti/adk as external consumer
  • IP/license blocker resolved
  • Single-author project — no dedicated team or community
  • Audit trail is a stub (issue #67), not wired through
  • Kubernetes operator is a skeleton (plain manifests today)
  • No observability (Prometheus/Grafana TBD)
  • FIPS delegated but not validated end-to-end
  • Genuine prototype — production hardening needed

DELIVERY ROADMAP

Feature Phasing

Incremental delivery aligned with RHOAI release cadence (~3 months per release).

SHORT-TERM — RHOAI 3.6 (NOV 2026)

Limited Tech Preview

  • Memory as independent MCP service (decoupled from Responses API)
  • Record-level scope isolation (user / project / org)
  • Basic audit logging (append-only write events)
  • MCP tool surface (read / write / search / curate)
  • PII scanning on write path
  • Hybrid retrieval (vector + BM25 at minimum)
  • PostgreSQL + pgvector single-substrate backend
  • OLM-managed operator (extend Feast operator or build new)
LONG-TERM — RHOAI 3.7+ (2027)

GA Target

  • Full provenance tracking (source_agent, identity, lineage per record)
  • Inspectable compaction (human-readable summaries, not opaque)
  • Contradiction detection with curator workflow
  • GDPR Article 17 erasure across all memory tiers
  • Multi-agent memory coordination + scope promotion
  • Memory versioning with full history chains
  • Observability (Prometheus metrics, Grafana dashboards)
  • FIPS end-to-end validation
  • Sleep-time consolidation with governance gate
  • KV-cache-aware memory ordering (Red Hat differentiator)

EU AI Act enforcement: August 2026. Audit trail and inspectable compaction are compliance-gating features. A 3.6 Tech Preview can ship without full compliance, but GA must have them. Neither OGX nor MemoryHub ships a working audit trail today — this is the single highest-severity gap (Q-G7).

PATHS FORWARD

Three Proposed Architectures

Each path has different champions, different trade-offs, and different assumptions about where value concentrates. All are proposals under evaluation for the July architecture review.

PATH A

Feast-Led Full Stack

Champions: Jonathan Zarecki (PM), Umberto Manganiello (Eng), Feast team

Feast owns the full memory stack — substrate, governance, and retrieval — via the 3-phase plan (A: Discovery/MCP → B: Retrieval/MemoryFeatureView → C: Write/Mem0 integration). Mem0 provides the cognitive layer (extraction, contradiction detection) while Feast enforces the data boundary. “Mem0 owns the cognitive logic; Feast owns the data boundary.”

Strengths
  • Feast GA in 3.4 — production RBAC, OLM, PIT correctness ship today
  • Dedicated Red Hat team + upstream community
  • 3-phase roadmap already designed with clear phasing
  • Cell-level masking for PII governance at the storage layer
  • feast init -t agent scaffolding for developer experience
Risks & Gaps
  • Feature store → memory service is a significant pivot
  • 3 deltas needed (record-level scope, audit log, provenance)
  • Phase C depends on Mem0 supporting pluggable storage backends
  • No memory-specific intelligence built yet (all on roadmap)
  • Inherits Mem0 open-core risk if Mem0 is the cognitive layer
PATH B

MemoryHub-Led Independent Service

Champions: Sanjeev Rampal (OCTO/ET), Wes Jackson (author)

MemoryHub becomes the standalone memory service, decoupled from both OGX and Feast. It already covers intelligence (PII scan, dedup, contradiction detection) and governance (6-tier scope, versioning, MCP server) in a single package. Exposed as MCP tools through MCP Gateway. IP/license blocker now resolved — Wes Jackson transferring full ownership.

Strengths
  • Purpose-built for agent memory — most features already implemented
  • 6-tier scope model, contradiction detection, PII curation, versioning
  • MCP server with 3 profiles (compact/full/minimal)
  • UBI9 images, air-gap deployable, kagenti/adk SDK consumer
  • IP/license fully resolved
  • Could ship as limited Tech Preview in 2026
Risks & Gaps
  • Single-author project — bus factor of 1, no team or community
  • Audit trail is a stub (not wired through)
  • Kubernetes operator is a skeleton
  • No observability, no FIPS validation
  • Genuine prototype — production hardening needed
  • Extraction less sophisticated than Mem0
PATH C

Composable — Best of Each

Proposed synthesis — uses each project for what it’s best at

Compose the layers independently: Feast for governance infrastructure (RBAC, PIT, OLM), MemoryHub patterns for memory-specific intelligence (scope tiers, contradiction, PII, versioning), and Mem0 as a pluggable backend compatibility target. No single project owns everything — each contributes where it’s strongest.

Strengths
  • Feast’s production governance + MemoryHub’s memory-native design
  • No single point of failure — reduces risk from any one project
  • Mem0 compatibility gives customers a migration path
  • Each team contributes from their area of expertise
  • Survives OGX deprecation, framework changes, and market shifts
Risks & Gaps
  • Integration complexity — composing 3 projects is harder than owning 1
  • Unclear ownership boundaries between Feast team and MemoryHub
  • Requires cross-team coordination (Feast team + ET + PM)
  • May result in “designed by committee” if boundaries aren’t crisp
  • Longer path to a unified developer experience

Common ground across all paths: MCP as the transport protocol. PostgreSQL + pgvector as the storage substrate. Memory decoupled from the Responses API. Exposed through MCP Gateway for per-tool authz and audit. The July architecture review should evaluate which path best balances delivery speed, team capacity, and enterprise readiness.
