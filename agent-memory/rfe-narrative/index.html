ACT ZERO · THE DESTINATION

Where this ends: memory as a platform promise

Every piece below exists to reach one end state. Hold this picture while reading the waves.

An AI engineer builds an agent in whatever framework their team uses. The agent remembers: it keeps working context during a task, recalls past episodes, accumulates facts, and refines learned procedures, all through one platform capability the engineer never had to build. When the team changes frameworks, the memory comes along. The agent gets better with use, because memory captures, consolidates, and curates itself, weighting what worked and what failed.

And every one of those memories is governed. Each record carries a scope and an origin. Writes are screened before anything sensitive can enter. Sharing across teams is a deliberate, recorded act, never an accident; conflicting memories are flagged, not silently absorbed. Engineers see what an agent remembered, and why it recalled what it recalled, inside Gen AI Studio. Platform engineers and compliance reviewers govern the whole estate from AI Hub: every store, every agent binding, every quarantined write, every line of the audit trail.

The result is bigger than any single agent: the organization accumulates shared, trusted, inspectable knowledge across its agents. It runs self-hosted and air-gapped, on storage the environment chooses, on models the customer hosts. Enterprise governance, Kubernetes-native deployment, and an open interface in one offering: the position no hyperscaler, database vendor, or memory startup occupies.

One governed service

Its own service, coexisting with the inference stack but dependent on none of it. Four memory access patterns over one store, on configurable backends.

Portable across frameworks

Switching models is cheap; switching memory is not. The service makes memory survive the switch, from any major framework or harness.

Trusted when shared

Scoped, screened, attributed, conflict-aware, and auditable. Shared memory that a security reviewer can approve, not a poisoned well.

Visible and operable

Engineers steer memory in Gen AI Studio. Admins govern the estate in AI Hub. Nothing important happens only behind an API.

ACT ONE · SIX RFEs

Wave 1: memory that works, governable from day oneProposed landing: 3.6 Dev Preview

The foundation wave delivers a working memory service and refuses to ship it ungoverned. Dev Preview is deliberate: it preserves the freedom to swap components before Tech Preview support, documentation, and sales obligations attach.

RHAIRFE-2630

Framework-agnostic agent memory service

The core substrate: one governed store exposing working, episodic, semantic, and procedural access patterns, on operator-configurable backends (vector database and file store minimum, extensible to graph), with hybrid retrieval and a framework-agnostic API. Self-hosted, air-gap capable.

Drives the end goal: everything else in the program hangs off this service; portability and backend freedom start here.

RHAIRFE-2631

Memory as governed MCP tools

Store, search, and recall become MCP tool calls: discoverable in the platform catalog, correctly scoped to user and project even over stateless connections, governed like every other MCP server.

Drives the end goal: one integration serves every MCP-capable harness, and memory joins the governed tool estate instead of sitting beside it.

RHAIRFE-2632

Built-in memory in Gen AI Studio (interim)

The platform's existing Responses-layer memory tool, surfaced in the Gen AI Studio playground: explicit "remember this" plus implicit capture, per-user isolation, view and delete. Deliberately allowed to run alongside the comprehensive service.

Drives the end goal: users feel memory now, demos stop starting cold, and real usage feedback de-risks everything that follows.

RHAIRFE-2633

Record-level scope isolation

Every memory record carries a scope; reads return only what the caller's scope permits; teams on a shared cluster cannot read each other's memories, demonstrably.

Drives the end goal: trust starts at the first record. Isolation shipped with the first preview is the floor everything shared is later built on.

RHAIRFE-2634

Sensitive-data screening on the write path

Memory writes are screened for PII, secrets, and sensitive patterns; policy decides whether a flagged write is blocked, quarantined, or logged; admins review and remove what gets caught.

Drives the end goal: regulated adoption depends on nothing toxic entering memory in the first place. Screening at the write path is cheaper than remediation ever is.

RHAIRFE-2635

Agent-memory write auditability

Every write is recorded: who, what, when, into which scope, in a log that administrators can review and authorized users can export for compliance.

Drives the end goal: the seed of the full audit trail. Compliance reviews get their first real artifact two releases before GA.

The story after Wave 1 lands

An agent on OpenShift AI remembers across sessions, from at least three major frameworks, on storage you choose, including fully disconnected clusters. Every memory is scoped, screened, and attributed from the very first record. Anyone can feel it working in the Gen AI Studio playground today. And because this is Dev Preview on purpose, any component can still be swapped before bigger obligations attach.

ACT TWO · EIGHT RFEs

Wave 2: from working memory to trusted, visible memoryProposed landing: 3.7 Tech Preview

The second wave turns a capable service into something teams rely on: memory that builds itself, is shared deliberately, and is visible to every persona that has to trust it. This is the wave that sets up the Summit 2027 story.

RHAIRFE-2636

Automatic memory creation and curation

Relevant information is captured automatically during interactions; memories consolidate over time; episodic records carry outcome signal so failures are not repeated; curation quality is measurable and the approach is pluggable.

Drives the end goal: "the agent gets better with use" becomes real. Manual-only capture never survives contact with production.

RHAIRFE-2637

Shared memory: tiers, conflicts, provenance

Memory scoped to user, project, role, and organization tiers; sharing across a boundary is a deliberate, recorded promotion; conflicting memories are flagged with both origins; every record traces back to what created it.

Drives the end goal: the organizational-memory step. Sharing without conflict handling and provenance would be a poisoned well; this wave couples them.

RHAIRFE-2638

Inspectable context engineering

Long-running sessions hold up past context limits; compaction produces a human-readable record of what was kept, summarized, and dropped; relevant memories are assembled into context automatically; token efficiency is measured at equal quality.

Drives the end goal: the read side of inspectability. A reviewer can reconstruct what the agent knew when it acted, and the cost story gets a number.

RHAIRFE-2639

Memory as a governed registry asset

Memory services register in the AI Asset Registry with ownership and lifecycle state; agent-to-memory bindings are recorded and queryable; asset-level governance actions apply to memory like any other asset type.

Drives the end goal: one control plane for models, prompts, MCP servers, and now memory. Decommissioning a store stops being a blind operation.

RHAIRFE-2640

Memory visibility in Gen AI Studio

AI engineers see the stores available to them, attach one to an agent or session without code, inspect what the agent remembered and what it recalled in a given exchange, and correct records, all inside the studio.

Drives the end goal: trust at the engineer level. Memory nobody can see gets turned off; memory you can steer stays on.

RHAIRFE-2641

Memory governance console in AI Hub

One admin surface: inventory of services and stores, agent bindings, quarantine review and disposition, write-log review and export, scope-policy administration. Fully functional in disconnected deployments.

Drives the end goal: governance becomes operable. Security reviews run against a console, not a curl command.

RHAIRFE-2642

Harness and framework integration packs

Ready-made, versioned, maintained integrations for the major agent frameworks and harnesses: platform memory works out of the box, participating in each harness's native context assembly rather than requiring glue code.

Drives the end goal: the framework-agnostic promise at adoption time. Portability that requires custom plumbing is not portability.

RHAIRFE-2643

Memory effectiveness on smaller self-hosted models

Memory that works reliably with the smaller models sovereign and on-premise customers actually run, with shipped guidance and assets, a measurable effectiveness bar, and disconnected usability.

Drives the end goal: the differentiator. The customers who chose self-hosted AI are exactly the ones the hyperscaler memory offerings cannot serve.

The story after Wave 2 lands

Memory now builds and curates itself, is shared across teams deliberately and with provenance, and is visible everywhere it matters: engineers steer it inside Gen AI Studio, admins govern the whole estate from AI Hub, and it works out of the box from the harnesses teams already use, on the models they host themselves. Long-running agents stay coherent and reviewable. This is the Tech Preview that carries the Summit 2027 demo.

ACT THREE · DIRECTIONAL

Wave 3: production trustProposed landing: 3.8 GANot yet filed

The hardening act. These four are deliberately unfiled: the earlier waves will teach us what GA really requires, and these RFEs get written with that evidence in hand rather than guessed at now.

Append-only audit trail + erasure

Every read and write in an immutable, inspectable trail, with erasure primitives. The GA gate that regulated customers have already named as their production condition.

Operator, observability, FIPS

OLM-managed lifecycle, metrics and dashboards for SRE and support, FIPS validation end to end. The difference between a prototype and a Red Hat product.

Adversarial memory defense

Injection patterns detected on write, poisoned memories quarantined, shared tiers protected. Memory treated as the attack surface it is.

Memory quality benchmarking

Recall quality measurable against standard benchmarks before production rollout. The numbers that prove the sophisticated architecture earns its complexity.

The story after Wave 3 lands

Governed agent memory is production-ready for regulated environments: fully audited, erasable on demand, defended against poisoning, quality-measured, and operable by any OpenShift team. The destination from the top of this page is no longer a narrative. It ships.
