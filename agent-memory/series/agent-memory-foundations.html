Seven Cross-Solution Gaps

Despite 16+ solutions, seven gaps remain unfilled across the entire landscape:

Gap 1: Governance + K8s + Air-Gap + Open

No solution combines all four. This is the whitespace.

Gap 2: Framework Portability

Memories are locked to the framework that created them. No standard transfer format.

Gap 3: Inspectable Compaction

Most compaction is opaque. EU AI Act requires human-readable output.

Gap 4: Multi-Session at Scale

Hardest benchmark category across all systems. Mem0: 48.6% at 10M tokens.

Gap 5: One Substrate or Many?

The central design question. Oracle says one; multi-store advocates say it depends on scale.

Gap 6: Quality Metrics

No standard metric for compaction quality. Benchmarks measure retrieval, not write-path quality.

Gap 7: Consolidation Risk

$62M+ into 5+ startups with near-identical approaches. Adopting one risks picking a non-survivor.

INDUSTRY LENSES

Three Lenses for Decomposing the Space

The industry has converged on three analytical lenses for understanding agent memory. Each answers a different question — and a sound product architecture needs all three.

What is stored?

The CoALA 4-Type Model

The dominant academic and vendor taxonomy (Princeton, 2023). Used by Oracle, IBM, Mem0, and most major frameworks.

  • Working — in-context, session-bound
  • Episodic — what happened
  • Semantic — durable facts
  • Procedural — how to do things
Oracle’s key claim: four access patterns over one substrate, not four systems.
Who can access what?

Scope & Governance Models

Orthogonal to the type model — classifies access control, not content type. Any memory of any CoALA type can sit at any scope tier.

  • Mem0: user / agent / run / app dimensions
  • Oracle: per-record scoping fields
  • MemoryHub: 5–6 hierarchical tiers
Implementations vary; the need for record-level scope is universal.
What data architecture?

Memory vs. RAG Divide

A two-way split by write path. Different data architectures with different governance needs. Complementary, not alternatives.

  • Memory — append-heavy, agent-written, consistency-critical (database concern)
  • RAG — batch-indexed, read-only, retrieval-optimized (search concern)
Conflating them in one feature produces an architecture that serves neither well.
Synthesis

A product architecture needs all three lenses: the CoALA model defines the internal vocabulary (what the substrate stores and how it’s accessed). The scope model defines governance (who can read/write). The Memory-vs-RAG divide is the architectural guardrail (don’t conflate a database concern with a search concern).

SOLUTION APPROACHES

How Could This Be Built?

Five broad approaches exist for delivering a governed agent memory capability. Each has distinct trade-offs on time-to-market, governance depth, and long-term flexibility. The implementation choice is an engineering decision — what matters at the strategy level is understanding the trade-offs.

Option A

Build Fresh

Purpose-built from scratch. Maximum design control; highest cost and longest timeline.

+ Full control over architecture and governance
− Longest time to customer; highest investment
Option B

Productize an Existing Prototype

Take an existing implementation (MemoryHub, or others) and close the productization gaps.

+ Faster to first customer; real code to evaluate
− Inherits existing design decisions and gaps
Option C

Extend an Existing API Surface

Build governance on top of an existing memory-capable API (e.g., the Responses API surface).

+ Leverages existing primitives; governance-focused effort
− Constrained by the underlying API’s design
Option D

Upstream Contribution

Contribute memory primitives and governance hooks into upstream projects and consume them.

+ Standards-aligned; community-validated; Red Hat’s traditional model
− Timeline not Red-Hat-controlled; upstream priorities may diverge
Option E

Partner with a Memory Vendor

Integrate a third-party memory solution (Mem0, Zep, etc.) as the primary substrate.

+ Fastest to market; proven implementations
− Consolidation risk ($62M+ into 5+ startups); governance gaps remain Red Hat’s to close

In practice, most enterprise platforms combine approaches — different options suit different subsystems and different phases of delivery. The choice depends on governance requirements, timeline, and what already exists in the platform.

KEY DECISIONS

What Must Be Decided

The research surfaces decisions that block feature scoping. These are grouped by what they unlock.

Blocks Everything

Decomposition & Scope

  • Accept the decomposition? 3 subsystems + 2 cross-cutting, or retain 3 peer areas?
  • Scope boundary? Memory substrate only, or include Agent Knowledge?
  • Confirm the substrate verdict? One substrate for 4 memory types; separate subsystem for org knowledge?
Highest Severity

Governance & Compliance

  • Audit trail delivery plan? Neither candidate ships one. EU AI Act enforcement Aug 2026.
  • Memory as a registry asset? Is the service governed? Are individual records governed?
  • Actor-chain RBAC? How does shared memory enforce lowest-permission when Agent B reads on behalf of Agent A?
Cross-Domain

Boundaries & Standards

  • Procedural memory boundary? Where does the skills registry end and the memory substrate begin?
  • Knowledge Sources extension? Is Agent Knowledge a new asset type or an extension of the existing one?
  • Standards program? MCP Memory Convention, A2A AgentCard memory binding, AAIF project — what timeline?
